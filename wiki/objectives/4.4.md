# 4.4 — Security Alerting and Monitoring

CompTIA description: *"Explain security alerting and monitoring concepts and tools."*

Monitoring is the eyes-and-ears layer. The objective splits cleanly into **concepts** (what you monitor, what you alert on, how you reduce false positives) and **tools** (SCAP, SIEM, antivirus, DLP, SNMP, NetFlow, vulnerability scanners). Know what each tool's job is and how it differs from its neighbors.

---

## Monitoring concepts

### What gets monitored

| Source | What it reveals |
|---|---|
| **Computing systems** | Authentications, login locations, services running, software versions, backups |
| **Applications** | Availability, traffic volumes, anomalous request patterns |
| **Infrastructure** | VPN sessions, firewall traffic, IPS detections, remote-access flows |

### Why centralize logs

Every system logs in its own format. Looking at the firewall log and the VPN log and the application log separately wastes time and misses correlations. Centralizing solves both problems.

### The SIEM as the consolidator

A **SIEM** (Security Information and Event Manager) is the central log database and reporting engine. It pulls logs from firewalls, switches, routers, servers, applications, IPSes, and stores them in one searchable place.

Why this matters:

- **Cross-source reporting** — one report can join VPN auth events with application activity and firewall actions.
- **Correlation** — link a login on one system to data movement on another.
- **Forensics** — long-term storage lets you reconstruct events months later (the IBM 2022 average to detect and contain a breach is ~9 months — backups and SIEM history must cover that window).

### Alerts vs reports

| | Reports | Alerts |
|---|---|---|
| **Timing** | Scheduled (daily / weekly / on-demand) | Real-time |
| **Audience** | Human reviewers | Automation, SOC, on-call |
| **Use case** | Trend analysis, compliance, weekly review | Immediate response to active events |

Alerts can be ad-hoc reports for hypothetical "what if X happens" planning, or actionable real-time signals (SMS, email, SOC dashboard ping).

### False positives, false negatives, and tuning

Same problem as in vulnerability scanning: minimize false negatives (missed real events) without drowning the team in false positives. Tuning is iterative — start broad, narrow as you learn what's noise.

### Quarantine as a common alert response

A common automated response to an alert is **quarantine** — isolate the affected host so it can't talk to other systems or exfiltrate further. The system stays up for forensic analysis but is bounded.

## Monitoring tools

### SCAP — the common language

**SCAP** (Security Content Automation Protocol) is a NIST-maintained protocol that standardizes how vulnerabilities are named and described across tools. Without SCAP, a NGFW, IPS, and vulnerability scanner each call the same flaw three different things.

With SCAP:

- Tools speak the same vocabulary
- Detection + ticketing + patching can be chained into automation
- Scan output from one tool feeds remediation in another with no manual translation

> **Exam trap:** SCAP isn't a tool — it's the protocol/format the tools use. Think of it like a wire protocol that lets a vulnerability scanner, a SIEM, and a patching system agree on which CVE they're talking about.

### Benchmarks — the reference standard

A **benchmark** is a published, vetted security configuration for a platform. The most widely cited library is the **CIS Benchmarks** at cisecurity.org. Use a benchmark to define what "secure baseline" actually means for a given OS, mobile platform, application, or cloud service.

### Agent-based vs agentless checks

| Method | Description | Trade-off |
|---|---|---|
| **Agent-based** | Persistent software installed on the system, always running | Continuous monitoring, but agents need maintenance and updates |
| **Agentless** | Runs in memory during login/connection, removes itself after | No install burden, but only runs when triggered — not continuous |

### Antivirus / antimalware

Traditional signature-based detection of known viruses, worms, Trojans, ransomware, fileless malware. Modern endpoint security has largely shifted to **EDR / XDR** (objective 4.5), but AV remains foundational.

### Data Loss Prevention (DLP)

DLP looks for and blocks sensitive data — SSNs, medical records, credit card numbers — in motion or at rest. Lives in three places:

- **Endpoint DLP** — on the workstation (data in use)
- **Network DLP** — on the wire (data in motion); often integrated with NGFW
- **Cloud / storage DLP** — scanning files at rest in SaaS / object storage

### SNMP — Simple Network Management Protocol

The classic device-monitoring protocol.

| Mechanism | Description |
|---|---|
| **Polling** | Management station queries device on a schedule (UDP **161**) |
| **Trap** | Device proactively notifies management station of an event (UDP **162**) |
| **MIB** (Management Information Base) | Database of metrics on the device |
| **OID** (Object Identifier) | The numeric address of a specific metric within the MIB |

> **Exam trap:** SNMP polls run on UDP **161**; SNMP traps run on UDP **162**. The exam expects both. Polling is pull-based and periodic; traps are push-based and event-driven.

### NetFlow — traffic-flow telemetry

Higher-level than SNMP. NetFlow records describe **traffic flows** — source/destination, ports, bytes, application — letting you see who talks to whom and how much.

| Component | Role |
|---|---|
| **Probe / exporter** | Built into a switch/router, or external; sees the traffic via SPAN or tap |
| **Collector** | Receives flow records, stores them, generates reports |

Typical output: top 10 conversations, top 10 endpoints by IP/FQDN, traffic by application. Great for understanding normal patterns so anomalies stand out.

> **Exam trap:** Don't confuse SNMP (device metrics like CPU and interface utilization) with NetFlow (traffic flow telemetry — who talked to whom and how much). They answer different questions and are often used side by side.

### Vulnerability scanners

Listed here as a *monitoring* tool: regular scheduled scans monitor for new vulnerabilities introduced by configuration drift, new software installs, or newly disclosed CVEs affecting existing systems. See [[objectives/4.3|objective 4.3]] for the analysis side.

## How to read scenario questions

1. **Concept or tool?** "How do we reduce false positives?" → concept. "Which protocol uses UDP 162 for traps?" → tool (SNMP).
2. **What's the source?** Endpoint (AV, endpoint DLP), network (NetFlow, network DLP, IPS), device metrics (SNMP), all-of-the-above (SIEM).
3. **Push or pull?** SNMP polling is pull; SNMP traps are push. NetFlow is push.
4. **Same language across tools?** That's SCAP.
5. **"All my logs in one place"?** That's a SIEM.
