# 4.9 — Data Sources for Investigation

CompTIA description: *"Given a scenario, use data sources to support an investigation."*

This objective is about **knowing what's in each log** so you can pick the right source for the question being asked. Eight categories of data feed an investigation, and they typically roll up into a SIEM that supports cross-source correlation, dashboards, and reports.

---

## The investigator's data landscape

| Source | What you find there |
|---|---|
| **Firewall logs** | Allowed/blocked flows, src/dst IP+port, NGFW app and URL category |
| **Application logs** | Per-app events (Windows Event Viewer → Application; Linux/macOS → `/var/log`) |
| **Endpoint logs** | Login/logoff, process events, password changes, directory service activity |
| **OS security logs** | Brute force, file changes, authentication, service state changes |
| **IPS/IDS logs** | Class, priority, source/destination of detected attacks (Snort-style) |
| **Network device logs** | Routing changes, switch/router auth errors, SYN attacks blocked |
| **Metadata** | Email headers, EXIF/GPS, browser fingerprints, document author info |
| **Vulnerability scan logs** | Missing AV/firewall, misconfigurations, unsupported OS, world-readable shares |

All of them roll up to the SIEM, where correlation and visualization happens.

---

## Firewall logs

The single richest network-level source. The firewall sees every flow that crosses the perimeter (or zone boundary), so its log can reconstruct most network-level activity after the fact.

| Field | What it gives you |
|---|---|
| Timestamp | When the flow occurred |
| Source / destination IP | Who talked to whom |
| Source / destination port | What service/protocol |
| Disposition | Allowed or blocked |
| Application (NGFW only) | L7 identification — e.g., "Twitter post" vs "Twitter view" |
| URL category (NGFW) | Where users were going |

Each line is a single flow. Reading volume per host, repeated denials, or unexpected destinations is investigative gold.

> **Exam trap:** Traditional L4 firewall logs give you port/IP/disposition. **NGFW** logs give you application *and* URL category in addition. If the question turns on "this app on port 443," it's NGFW.

---

## Application logs

On **Windows**: Event Viewer, **Application** section (the Security and System sections are separate).
On **Linux / macOS**: text files under **`/var/log`** — `syslog`, `auth.log`, `messages`, application-specific subdirectories.

Useful for app-specific errors, crashes, config changes, plugin loads, and any custom events the app emits.

---

## Endpoint logs

The endpoint sees what happens inside the user's session and on the device itself. Categories:

- **Authentication events** — login, logoff, lockout.
- **System events** — service start/stop, driver loads, time changes.
- **Process events** — what was started, by whom, with what arguments (especially with EDR augmentation).
- **Management events** — password changes, MFA enrollments, BitLocker key escrows.
- **Directory services** — group changes, GPO applications.

Combined with network logs in the SIEM, endpoint data closes the loop: "This IP made an outbound connection at 02:14" + "This user account started this process at 02:13" → a single correlated event.

---

## OS security logs

Distinct from application logs — these are the OS's own security-relevant events. Key things to look for:

- **Brute-force attempts** — repeated authentication failures from the same source.
- **File-integrity events** — unexpected changes to system files.
- **Service-state anomalies** — a security service (AV, EDR, audit) being disabled when no admin should be touching it.
- **Privilege escalations** — accounts being added to admin groups.

A single suspicious event — "the audit service was just disabled" — can fire an immediate alert.

---

## IPS / IDS logs

Modern deployments fold IPS/IDS into the NGFW, but the log format is similar to standalone tools like **Snort**:

| Field | Example |
|---|---|
| Timestamp | `2026-05-21 03:14:22` |
| Class | `Possible denial of service attack — SYN flood` |
| Priority | `2` |
| Source IP/port → Destination IP/port | `198.51.100.7:48224 → 10.0.0.42:443` |

IPS/IDS logs identify *what kind of attack* — they map traffic to known attack signatures. NGFW firewall logs identify *what flow* — they're connection-level. Both rolled up to the SIEM give you the most complete picture.

---

## Network infrastructure logs

Switches, routers, wireless controllers, VPN concentrators all log:

- **Routing changes** — new routes appearing, BGP/OSPF anomalies.
- **Authentication errors** — failed logins to manage the device.
- **L2/L3 attack detection** — SYN flood thresholds, MAC flooding, rogue DHCP.

Example log line: *informational — TCP SYN attack identified on gigabit interface 8; SYN traffic to local system automatically blocked for 60 seconds*. Network gear catching attacks at the edge gives you fast warning.

---

## Metadata — the data about the data

Often overlooked, often the most revealing source.

| Document type | Metadata that matters |
|---|---|
| **Email** | Full headers — sending server IP chain (`Received:`), SPF/DKIM/DMARC results, signing info, message-ID, true sender behind display name |
| **Image** | EXIF — camera model, software, GPS coordinates of capture |
| **Browser** | User agent (OS, browser, version), client IP, accept-language |
| **Word/Excel/PDF** | Author, organization, last-modified time, sometimes hidden tracked changes or comments |

> **Exam trap:** "Determine where an email actually came from" → email headers, not the From: line. The displayed sender is trivially spoofed; the Received chain and SPF result aren't.

---

## Vulnerability scan logs

Vulnerability scans don't just report CVEs. The scan output also flags configuration problems:

- Hosts with **no firewall** enabled.
- Hosts with **no antivirus** or anti-spyware.
- **Misconfigured services** — guest accounts enabled, NFS shares world-readable, anonymous SMB access.
- **Unsupported operating systems** still in production.
- Known CVEs in installed software (once scanner signatures are updated).

Reviewing vuln-scan logs alongside firewall and endpoint logs lets an investigator both find what's wrong and check whether anyone exploited it before you found it.

---

## SIEM consolidation

The Security Information and Event Manager pulls all of these sources into one searchable, correlatable store. What it adds on top:

### Automated reports

Generated on a schedule and emailed to stakeholders. Be specific about what you want — generating a report from terabytes of data can take many hours, and unread reports are wasted compute.

> **Exam trap:** Messer's standing observation — reports get generated and then ignored. The exam may test the *value* of reports, but the practical reality is that dashboards get watched and emailed reports often don't.

### Dashboards

At-a-glance summaries: active firewall rules, current warnings, top talkers, anomalous user behavior. Designed to render instantly — they show the *current* picture, not long-term aggregates. Common in the SOC NOC display wall.

### Cross-source correlation

The real reason for the SIEM. A firewall blocking outbound traffic to a known C2 IP, plus an endpoint executing an unsigned binary at the same timestamp, plus a vuln-scan log showing that endpoint as unpatched → one correlated incident instead of three orphan events.

## Packet captures

When logs aren't enough, capture at the packet level. **Wireshark** is the canonical tool for capturing from a host's interface; many switches, routers, and firewalls can capture packets internally and export them.

A Wireshark capture has two main panes:

| Pane | What it shows |
|---|---|
| **Summary (top)** | Packet-by-packet list — timestamp, src/dst, protocol, info (e.g., `HTTP GET /index.html`) |
| **Detail (bottom)** | Layered breakdown of the selected packet — Ethernet header, IPv4 header, TCP header, HTTP payload |

Captures give you everything — every byte that crossed the wire. Useful for understanding exactly what an attacker did, what was exfiltrated, or how a protocol exchange went wrong.

---

## How to read scenario questions

1. **What artifact does the question describe?** Headers → metadata. SYN-flood detection → IPS or network-device log. Allowed/blocked flow → firewall. Login failures → endpoint or OS security log.
2. **Network-level or host-level?** Network-level → firewall, IPS, network gear. Host-level → endpoint, OS, application logs.
3. **One source or multiple?** If the question requires correlation across types → SIEM is the answer.
4. **Logs or packets?** "What kind of attack" → logs. "Exact bytes transferred" → packet capture.
5. **L4 or L7?** Port/IP only → traditional firewall log. Application/URL → NGFW.
