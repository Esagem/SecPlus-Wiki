# 4.3 — Vulnerability Management

CompTIA description: *"Explain various activities associated with vulnerability management."*

A continuous loop: **identify** vulnerabilities, **analyze** them in context, **decide** how to remediate, **verify** the fix. Five sub-areas cover it — scanning, threat intelligence, penetration testing, analysis, and remediation.

---

## The lifecycle

| Stage | Activity |
|---|---|
| **Identify** | Vulnerability scans, SAST/DAST, threat intelligence feeds, bug bounty reports |
| **Analyze** | Validate findings, score severity (CVSS), filter false positives, prioritize by exposure factor |
| **Remediate** | Patch, segment, transfer (insurance), or accept (exception/exemption) |
| **Verify** | Rescan after patching; audit a sample to confirm coverage |

---

## Vulnerability scanning

A vulnerability scan **looks for the potential** of an attack without actually exploiting it. (If it exploits, it's a penetration test.)

### Scan types

| Scan | Looks for |
|---|---|
| **Port scan** | Open TCP/UDP ports — surfaces, not vulnerabilities directly |
| **Network vulnerability scan** | Versions and configs with known CVEs |
| **SAST** (Static Application Security Testing) | Source-code patterns (buffer overflow, injection-prone code, unsafe APIs) |
| **DAST / Fuzzing** | Runtime behavior on malformed input — exceptions, crashes, oddities |
| **Web app scan** | Vulnerabilities in deployed web applications |
| **Application / mobile-app scan** | Vulnerabilities in installed apps |

### External vs internal scans

External (from the internet looking in) shows what an outside attacker sees. Internal scans assume the attacker is already inside — increasingly relevant given insider threats and post-breach lateral movement. Mature programs run both.

### What SAST can't catch

Static analysis is pattern-based. It misses:

- Crypto misuse (using AES wrong, weak parameters)
- Authentication flow flaws
- Business-logic errors
- Issues that depend on runtime data

Pair SAST with fuzzing/DAST and human review.

### Trust the package source

Before installing software (especially Linux packages), verify it came from the upstream vendor and not a third-party mirror that may have injected malware. Test in a lab before production rollout.

## Threat intelligence

Knowing what's coming at you. Threat intel feeds inform what to patch, what to scan for, what to block.

### Sources

| Source | What it is | Cost |
|---|---|---|
| **OSINT** (Open-Source Intelligence) | Public discussion groups, government reports, social media, news | Free |
| **Proprietary / third-party** | Commercial intelligence services compiling across customers | Paid |
| **Information-sharing alliances** | E.g. **Cyber Threat Alliance (CTA)** — pool intel across member organizations | Membership |
| **Dark web** | Forums, marketplaces (stolen data, exploits) — go directly to the source | Access |

### Why third-party feeds add value

A commercial threat intel provider sees attacks across many customer networks at once. A trend visible across multiple enterprises can be alerted *before* it reaches yours.

### Dark web

Reachable only via specialized software (Tor, etc.). Monitor for your organization's name, leaked credentials, customer data, or tool/technique chatter relevant to your industry.

## Penetration testing

Pen testing **simulates an actual attack** — exploiting, not just observing.

### Reference: NIST SP 800-115

The *Technical Guide to Information Security Testing and Assessment*. A standard reference for pen-testing methodology.

### Rules of Engagement (RoE)

Written agreement *before* the test starts. Specifies:

- **When** testing is allowed (often outside business hours)
- **What** is in scope (specific IP ranges, systems) — and what is **out of scope**
- **How** sensitive findings are handled
- **Who** to contact in emergencies
- Test types: on-site physical, internal, external

> **Exam trap:** RoE is *not* the test itself — it's the document that makes the test legal and safe. Skipping the RoE makes the testing activity indistinguishable from an actual attack.

### The pen test arc

1. **Initial exploit** — find a vulnerability, use it, gain a foothold.
2. **Persistence** — back door, added account, modified credentials. Don't rely on the same exploit; the patch may close it.
3. **Lateral movement / pivoting** — use the foothold to reach other internal systems; set up relays/proxies.
4. **Privilege escalation** — try buffer overflows, password brute force, social engineering. *Note: an instability from an exploit attempt can crash production — the RoE keeps critical systems off-limits.*

### Bug bounties and the disclosure lifecycle

Vulnerability discovery → reported to vendor privately → vendor builds and tests a patch → patch released → vulnerability disclosed publicly → CVE entry. Weeks or months can pass between discovery and disclosure. **Bug bounties** are formal reward programs encouraging this responsible disclosure.

## Analyzing vulnerabilities

### False positive vs false negative

| Term | Meaning |
|---|---|
| **False positive** | Scanner reports a vulnerability that doesn't actually exist |
| **False negative** | Scanner *misses* a real vulnerability — much worse |

Low- and informational-severity findings are not false positives — they're real, just low priority. Don't dismiss them by that label.

> **Exam trap:** A false negative is more dangerous than a false positive. A false positive wastes triage time; a false negative leaves an unknown attackable hole. Up-to-date signatures minimize both.

### CVE and CVSS

| Identifier | Source | Purpose |
|---|---|---|
| **CVE** (Common Vulnerabilities and Exposures) | MITRE — cve.mitre.org | Unique ID per known vulnerability |
| **NVD** (National Vulnerability Database) | NIST — nvd.nist.gov | Synchronized with CVE; adds analysis and CVSS |
| **CVSS** (Common Vulnerability Scoring System) | FIRST | 0–10 numeric severity score (CVSS 2.0 and 3.x in current use) |

Most scanners cross-reference findings to CVE; you cross-reference CVE to NVD and the vendor's bulletin for full context.

### Exposure factor

A percentage of system / service impact if the vulnerability is exploited. A vulnerability that takes down a service entirely is 100% EF; one that degrades it half the time is 50%. EF combined with CVSS plus environmental context drives priority.

### Environment matters

A CVSS-9 vulnerability on a public-facing system needs immediate attention. The same vulnerability on an isolated lab system gets a different priority. Environment-specific factors:

- **Public exposure** vs internal-only vs air-gapped
- **User count and type** — many users vs few; external vs internal
- **Revenue impact** — does the system make money?
- **Exploit complexity** — easy attacks rise to the top

### Risk tolerance

How much risk the organization is willing to accept while testing a patch. Lower tolerance → rush the patch testing, accept some risk of breaking things. Higher tolerance → test thoroughly, leave the vulnerability open longer.

## Vulnerability remediation

### Patching — the primary remediation

Most vulnerabilities are fixed by installing a vendor patch. Patches usually arrive on a regular cadence (Patch Tuesday); critical / zero-day patches arrive **out-of-band** on whatever schedule the vendor releases them.

### Insurance

**Cybersecurity insurance** transfers financial risk. Typical coverage: revenue loss, recovery costs, phishing-driven fraud, lawsuit liability. Typical *exclusions*: intentional acts, fund transfers initiated by the policyholder. Ransomware's prevalence has driven much wider adoption.

### Segmentation

When patching isn't possible (yet), reduce blast radius:

| Technique | What |
|---|---|
| **VLAN segmentation** | Logical separation on the same physical switch |
| **Air gap** | Physical separation — no network connectivity at all |
| **Next-Generation Firewall between segments** | App-aware traffic control and visibility |

### Compensating controls

When you can't patch and can't fully segment, you compensate:

- Disable the vulnerable service entirely.
- Revoke user access to the application.
- Block the exploit path at the perimeter firewall.
- Add ACLs in routers or host-based firewalls.

These are stopgaps, not cures — patch when possible.

### Exception / exemption

A formal decision **not** to patch a specific system, made by a security committee or change-control board. Justified when the vulnerability has very limited exposure (e.g., requires local console access on a server inside a guarded data center) or the patch conflicts with critical software. Documented; revisited.

> **Exam trap:** Exceptions are *committee* decisions, not individual ones. One person doesn't get to accept a risk on the organization's behalf.

### Verification

After remediation, **rescan** to confirm:

1. The patch was applied where expected.
2. There are no systems that were missed.
3. The vulnerability really is closed.

Combine automated reports from the patching system with manual spot-audits. At scale (hundreds or thousands of systems), reporting is essential — by-system and by-vulnerability views, plus exception/error/pending-patch breakdowns.

---

## How to read scenario questions

1. **What phase?** Identify (scan), analyze (CVSS, EF), remediate (patch / segment / compensate / accept), verify (rescan).
2. **What kind of finding?** Scan output → triage for false positives. Critical CVSS on internet-facing service → highest priority.
3. **Can't patch?** Compensate (block at firewall, disable service) or segment (VLAN / air gap), with exception process if needed.
4. **Patched everything — done?** No: rescan to verify.
