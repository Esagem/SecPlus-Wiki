# 1.4 — Cryptographic Solutions

CompTIA description: *"Explain the importance of using appropriate cryptographic solutions."*

Cryptography is a sprawling topic. The exam tests four buckets:

1. **Keys** — symmetric vs. asymmetric, how they're generated, exchanged, escrowed.
2. **Primitives built from keys** — hashes, digital signatures, certificates, key exchange.
3. **Uses** — data at rest, data in transit, database encryption, blockchain.
4. **Tools and hardware** — TPM, HSM, KMS, secure enclave.

Master how each piece works and how they slot together; the exam mostly tests *which tool for which job*.

---

## The two foundations: symmetric vs. asymmetric

| Property | Symmetric | Asymmetric |
|---|---|---|
| Keys involved | One shared key | A mathematically related public/private pair |
| Encrypt with | The shared key | The recipient's *public* key |
| Decrypt with | The same shared key | The recipient's *private* key |
| Speed | Fast | Slow |
| Scales to many parties? | Poorly — every pair needs its own key | Yes — publish one public key per person |
| Other names | Secret-key, shared-secret | Public-key |

Real systems almost always use **both**: asymmetric crypto to securely exchange a symmetric session key, then symmetric crypto to encrypt the actual bulk traffic. You get asymmetric's scalability and symmetric's speed.

**Trap:** You cannot derive a private key from its public key. The math (large prime factorization, elliptic curves) is one-way. Anyone may safely hold the public key.

---

## Public Key Infrastructure (PKI)

The policies, procedures, hardware, software, and roles for creating, distributing, managing, storing, and revoking digital certificates and keys. PKI is the *infrastructure* — not a single product. A small company still has PKI; it's just less elaborate.

PKI binds an identity (a person, a server, a device) to a public key through a certificate signed by a trusted authority.

### Key escrow

Storing copies of private keys in a controlled third party so they can be recovered if the original holder is unavailable — departed employee, lost device, government access requirement. Controversial but sometimes required for availability or compliance.

**Trap:** Key escrow is not the same as a backup of your own keys. Escrow specifically means someone *else* holds a usable copy.

---

## Encrypting data at rest

Where the data sits unencrypted matters; the right technique depends on the granularity you need.

| Approach | Granularity | Examples |
|---|---|---|
| **Full disk encryption (FDE)** | Whole volume | BitLocker (Windows), FileVault (macOS), dm-crypt/LUKS (Linux) |
| **File-level encryption** | Single file or folder | EFS (Windows NTFS), third-party tools |
| **Transparent database encryption** | Whole database, symmetric key | SQL Server TDE, Oracle TDE |
| **Column-level (DB) encryption** | One column inside a table | Encrypt SSN column; leave name/ID readable for fast search |
| **Record-level (DB) encryption** | One row | Per-record symmetric keys |

Column-level avoids the overhead of decrypting the whole DB for every search; you only pay the cost when you actually access the sensitive column.

**Trap:** EFS is a *file-system* feature (NTFS). BitLocker is *volume*-level FDE. They're complementary, not interchangeable.

---

## Encrypting data in transit

| Technology | Used for |
|---|---|
| **HTTPS (TLS)** | Browser ↔ web server |
| **SSL/TLS VPN** | Remote-user-to-site VPN, client-based |
| **IPsec VPN** | Site-to-site VPN between two networks |

Both sides must agree on the **encryption algorithm** to communicate. DES and AES, for example, are mutually incompatible — you can't encrypt with one and decrypt with the other.

The algorithms themselves are **public knowledge** — that's a feature, not a bug. Public algorithms get scrutinized; their security depends on the secrecy of the *key*, not the secrecy of the math. Like a door lock: everyone knows how locks work; you still need the right key to open one.

---

## Key length and key stretching

Both defend against brute force.

| Defense | Mechanism |
|---|---|
| **Longer keys** | More possible values → more guesses needed. 128+ bits for symmetric; 3072+ bits for asymmetric (RSA) today. |
| **Key stretching** | Hash the key, hash the hash, hash the hash of the hash, etc. Each guess in a brute-force attack now requires the same chain — adds compute overhead per attempt. |

Key stretching is why password-hashing functions like PBKDF2, bcrypt, scrypt, and Argon2 are deliberately *slow*.

---

## Key Exchange

The problem: how do two parties agree on a symmetric key over an untrusted network?

### Out-of-band

Hand the key over via some channel that isn't the network. Courier, phone call, in-person handoff, separate medium. Slow but secure when feasible.

### In-band

Send the key (or generate it) over the same network you'll use to communicate. Two patterns:

| Pattern | How it works |
|---|---|
| **Asymmetric wraps symmetric** | Encrypt a symmetric session key with the recipient's public key, send. Recipient decrypts with their private key. |
| **Diffie-Hellman-style key exchange** | Each side combines its own *private* key with the other side's *public* key. The math makes both arrive at the same shared symmetric secret without that secret ever crossing the wire. |

Session keys generated this way are usually **ephemeral** — used once, discarded, replaced for the next session. Limits exposure if a key ever leaks.

**Trap:** In a DH-style key exchange, *nothing secret is sent over the network*. The shared key is derived independently on each side. That's the magic; the exam likes to test that you understand it.

## Encryption hardware

| Hardware | Scope | Purpose |
|---|---|---|
| **TPM (Trusted Platform Module)** | One device | Per-device key generation/storage, random numbers, full-disk encryption keys, secure boot |
| **HSM (Hardware Security Module)** | Many devices / data center | Cluster-scale key generation, storage, and hardware-accelerated crypto operations |
| **Key Management System (KMS)** | Org-wide | Centralized management of *all* keys (SSL/TLS, SSH, AD, BitLocker) with rotation and reporting. Often cloud-based. |
| **Secure enclave** | One device, separate processor | Hardware-isolated processor inside a phone/laptop, owns its own boot ROM and crypto keys, isolates sensitive operations from the main CPU |

A KMS is software/service. A TPM and HSM are dedicated hardware. A secure enclave is a *processor within* a device. The categories don't overlap.

**Trap:** TPM is for the local machine. HSM is for many machines from a centralized cluster. Don't mix them up.

## Hashing

A hash function takes any input and produces a fixed-length output (a **message digest** / **fingerprint**).

| Property | Hash | Encryption |
|---|---|---|
| Reversible? | No | Yes (with key) |
| Output size | Fixed | Roughly proportional to input |
| Changing input | Completely different output | Completely different output |
| Used for | Integrity, password storage, signatures | Confidentiality |

Tiny input changes produce wildly different hashes — that's the avalanche property.

### Collisions

Two different inputs producing the *same* hash. Should be astronomically rare. When found, the hash function is considered broken — **MD5 had a documented collision in 1996**, which is why MD5 is deprecated and SHA-256 (or stronger) is recommended today.

### Salting

A random per-user value appended to the input *before* hashing. Two users with the same password get different stored hashes. Defeats **rainbow tables** — precomputed input → hash lookup tables — because the attacker would need a separate table per salt.

**Trap:** Passwords should be **salted and hashed**, not encrypted. Encryption is reversible — leak the key and you leak every password. Hashing is one-way, and salting makes precomputed attacks useless.

---

## Digital signatures

Mechanism: hash the message, encrypt the hash with the *sender's private key*. The encrypted hash is the signature, attached to the message.

| Recipient action | What it proves |
|---|---|
| Decrypt the signature with the sender's *public* key → recover the original hash | The signer's private key was used (proof of origin) |
| Compute the hash of the received message and compare | The message wasn't altered (proof of integrity) |
| Both match | Non-repudiation: sender can't deny it; anyone with the public key can verify |

The message itself isn't necessarily encrypted — signing is an integrity + origin operation, distinct from encryption.

**Trap:** Signing uses the *sender's* keys. Encryption uses the *recipient's* keys. Reversing them is the most common signature exam mistake.

---

## Certificates

A digital certificate binds an identity (server, person, device) to a public key, and is **signed by a Certificate Authority** so anyone trusting the CA can trust the binding.

### X.509 standard

Universal certificate format. Includes:

- Serial number, version
- Issuer (the CA)
- Subject (the holder)
- Validity period
- Public key
- Signature algorithm + signature
- Optional extensions (SAN, key usage, CRL/OCSP pointers)

### The CA lifecycle

1. Requester generates a key pair.
2. Requester creates a **Certificate Signing Request (CSR)** containing their public key + identity info.
3. CSR is sent to the CA.
4. CA *validates the request* — confirming the requester really controls the named domain/server/identity. **This validation is what you pay the CA for, not the signature itself.**
5. CA signs the cert with its private key and returns it.
6. Requester installs the signed cert on the server.

### Trust chains: public vs. internal CAs

| Type | Trusted by | Used for |
|---|---|---|
| **Public CA** | All major browsers and OSes by default (hundreds in the trust store) | Public-facing websites, email |
| **Internal/private CA** | Only devices that have explicitly installed the CA's root cert | Internal apps, employee VPNs, device authentication |

To use an internal CA, the CA's own root certificate must be in every client's trust store. Once it is, internal certs work identically to public ones.

### Root of trust

The trusted anchor at the top of the chain — hardware (TPM, HSM, secure enclave), a CA, or firmware. Everything downstream borrows its trust from the root.

### Web of trust

Alternative to a central CA. Individuals sign each other's certificates. If you trust Alice and Alice trusts Bob's cert, you transitively trust Bob's cert. Used by PGP / GPG, not by browsers.

### Subject Alternative Name (SAN) and wildcard certificates

A cert may cover more than one hostname:

- **SAN** lists explicit additional names: `mail.example.com`, `vpn.example.com`.
- **Wildcard** uses an asterisk: `*.example.com` covers `www.example.com`, `mail.example.com`, etc.

Greatly simplifies cert management when one organization runs many services on the same domain.

### Revocation: CRL and OCSP

When a cert needs to be invalidated early (compromise, decommissioning, employee departure):

| Mechanism | How it works | Trade-off |
|---|---|---|
| **CRL (Certificate Revocation List)** | CA publishes a big list of revoked cert serial numbers; clients download and check | Lookup is a download of a potentially huge file |
| **OCSP (Online Certificate Status Protocol)** | Client queries the CA in real time about a specific cert | One small lookup per cert; load shifts to the CA |
| **OCSP stapling** | Server includes a recent OCSP response (digitally signed by the CA) directly in the TLS handshake | No client lookup at all; revocation status arrives with the cert |

**Heartbleed** (April 2014, OpenSSL vulnerability) is the textbook example: thousands of private keys had to be presumed compromised, all corresponding certs revoked, and replaced. CRLs and OCSP did the bulk-revocation work.

**Trap:** OCSP stapling is performed by the *web server*, not the client. The server periodically fetches and caches a signed OCSP response from the CA, then includes ("staples") it in every TLS handshake. Client side does no OCSP work.

## Obfuscation

Hiding information *in plain sight*. Different from encryption: anyone who knows how it was hidden can recover it without a secret key. "Security through obscurity" — useful as a layer, never as the sole defense.

### Steganography

Hiding data inside another file ("covertext"):

| Medium | Example |
|---|---|
| Image | Adjust low-order bits of pixels to encode a hidden payload |
| Audio | Encode data in inaudible frequency ranges |
| Video | Per-frame embedding |
| Network | Hide bytes across the sequence of TCP packets |
| Print | The yellow "machine identification code" dots laser printers add to every printed page |

Roots in Greek for *concealed writing*.

### Tokenization

Replace sensitive data with a meaningless **token** that maps back to the original only via a trusted lookup service. The token has no mathematical relationship to the original — it cannot be reverse-engineered.

Mobile payments are the canonical example: your phone never transmits your real credit-card number. Instead the merchant sends a one-time token to the token service, which performs the reverse lookup and authorizes the charge.

**Trap:** Tokenization is not encryption. The token cannot be "decrypted" — only the token service can resolve it. And once consumed, single-use tokens cannot be replayed.

### Data masking

Show partial data, hide the rest. Receipt prints `****-****-****-2512` instead of the full card. Customer service reps see only the last four digits. Different from tokenization: data masking is for *display*; tokenization is for *transit/storage*.

---

## Blockchain

A **distributed ledger** — many participants each hold a synchronized copy of the same transaction history. New transactions are bundled into a **block**, hashed, and broadcast to every participant.

| Property | How it works |
|---|---|
| Tamper evidence | Each block contains a hash. Change any past transaction → hash no longer matches → that copy is rejected by other nodes. |
| Decentralization | No single authority owns the ledger; consensus across copies is what defines truth. |
| Use cases | Cryptocurrency (Bitcoin), supply-chain tracking, digital identification, voting, transfer of digital assets. |

The hash is the integrity glue. Blockchain leans on hashing more than on encryption.

---

## How to read scenario questions

Ask: what is the *goal* the scenario describes?

- "Prevent anyone from reading the data" → confidentiality → **encryption** (symmetric for bulk, asymmetric to exchange the key).
- "Detect tampering" → integrity → **hash**.
- "Prove who sent it, with non-repudiation" → **digital signature**.
- "Bind an identity to a public key with a trusted third party" → **certificate** issued by a **CA**.
- "Cover many subdomains with one cert" → **wildcard** / SAN.
- "Cert is no longer trusted" → **CRL** or **OCSP**.
- "Need centralized key storage for a data center" → **HSM**.
- "Per-device crypto for one laptop" → **TPM**.
- "Manage all keys across many services" → **KMS**.
- "Hide credit card during transit, can't be replayed" → **tokenization**.
- "Show partial info on screen for a CSR rep" → **data masking**.
- "Hide data in an image" → **steganography**.
