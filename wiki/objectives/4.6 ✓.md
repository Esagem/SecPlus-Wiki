# 4.6 — Identity and Access Management

CompTIA description: *"Given a scenario, implement and maintain identity and access management."*

Four blocks fit together: **provision the identity** (IAM lifecycle, proofing, federation protocols), **decide what it can touch** (access-control models), **prove it at login** (MFA factors), and **keep the secret itself safe** (password security, passwordless, JIT). The exam loves to swap one protocol for another (SAML vs OAuth, MAC vs DAC vs RBAC, RADIUS vs EAP) — the distinctions matter more than the definitions.

---

## The IAM lifecycle

IAM is the full arc from account creation to deactivation. The right access for the right person at the right time — no more, no less.

| Stage | Action |
|---|---|
| **Onboarding** | Provision account, assign group membership, apply baseline permissions |
| **Day-to-day** | Update permissions on role change, log all access |
| **Promotion / transfer** | Recalculate group membership, *remove* old rights, add new |
| **Offboarding** | De-provision — deactivate or delete the account |

A user record carries a **name**, **attributes**, **group memberships**, and **application-specific permissions**. Provisioning and de-provisioning are the core operations — getting either one wrong leaks access.

> **Exam trap:** "Joined the security team last month and still has finance permissions" → de-provisioning failure on transfer, not just offboarding. The lifecycle has middle stages.

### Identity proofing

Before you hand out an account, you must verify the human is who they claim to be. Three steps:

| Step | Purpose |
|---|---|
| **Resolution** | Identify the candidate uniquely — name, employee ID, applicant ID |
| **Validation** | Capture credentials that only they should know or possess (password choice, security questions) |
| **Attestation** | Verify via authoritative evidence — government ID, in-person meeting, automated knowledge-based questions tied to credit history |

> **Exam trap:** Proofing is *before* the account exists. MFA is *every login after*. Don't conflate them.

## Authentication protocols

### SSO — single sign-on

Log in once; access every authorized resource for the duration of the session (often 24 hours). Avoids re-prompting at every printer, share, and app. Requires that the underlying authentication and authorization protocols *support* SSO — LDAP/Kerberos in Active Directory, SAML, OAuth/OpenID.

### LDAP — the directory

**Lightweight Directory Access Protocol.** A protocol for reading large directories of users, groups, and devices. Standardized by the **X.500** specification (ITU-T). Successor to DAP (Directory Access Protocol), which ran on OSI — LDAP runs on TCP/IP and is much more portable.

LDAP names objects as **distinguished names** built from attribute-value pairs:

```
CN=widget1,OU=marketing,O=Acme,C=US
```

`CN` = common name (the leaf — a device, user, or service). `OU` = organizational unit (a department). `O` = organization. `C` = country. The tree of containers and leaves is the **directory information tree**.

Active Directory is the most common implementation in Windows environments. RADIUS or TACACS+ may sit in front for AAA flows.

### SAML — federated browser-based SSO

**Security Assertion Markup Language.** Three-party flow for browser SSO to a third-party app:

```
[1] Client → Resource server: "I'd like to access this URL."
[2] Resource server → Client: "Not logged in — here's a SAML request. Take it to the auth server."
[3] Client → Auth server: presents credentials.
[4] Auth server → Client: SAML token (signed assertion).
[5] Client → Resource server: presents SAML token. Granted access.
```

Designed before mobile — works fine in a browser, awkward on a phone app constantly switching contexts. That's why OAuth showed up.

### OAuth + OpenID — mobile-friendly authorization

**OAuth is an authorization framework** — it decides *what* a logged-in user can access on a third-party resource. It does **not** authenticate the user; **OpenID Connect** is paired on top to do that.

Example: you visit Zapier and want it to read your Google Drive. Zapier redirects you to Google. Google authenticates you (via OpenID), then asks "Allow Zapier to view, edit, create, and delete files on your Drive?" You click allow. Google issues Zapier a token. Zapier now acts on your behalf with exactly that scope.

| | SAML | OAuth + OpenID |
|---|---|---|
| Designed for | Browser SSO between enterprises | Mobile and API access across services |
| What it does | Authentication assertion | OAuth = authorization, OpenID = authentication |
| Token format | Signed XML assertion | Bearer tokens (often JWT) |

> **Exam trap:** SAML is one protocol that handles authentication. OAuth handles *only* authorization — pair it with OpenID Connect for authentication. "Mobile app delegating limited access" → OAuth. "Browser SSO to a corporate app" → SAML.

### Federation

Network access without a local user database. A site accepts identities issued by a partner (Google, Twitter, Facebook, LinkedIn, an enterprise IdP). The site and the IdP set up a trust relationship in advance; any user with an account at the IdP can then log in to the site without creating a new one.

OAuth/OpenID and SAML are the protocols that *implement* federation. Federation is the architectural choice — protocols are the wire format.

### Interoperability decisions

Picking IAM protocols is constrained by what already exists. A new VPN concentrator that speaks LDAP slots into an existing AD. A new application that wants OAuth needs an authentication server you can wire to it. The decision rolls up to current infrastructure and future direction, not protocol popularity alone.

## Access-control models

After authentication, the system enforces *what* the identity may touch. Five models on the exam, all underpinned by one principle.

### Least privilege (the baseline)

Assign the rights necessary to do the job and nothing more. Default users do **not** get admin rights. If malware runs in a user's session, it inherits that user's limited privileges — containment by design.

### MAC — Mandatory Access Control

The **administrator** assigns clearances to users and labels to data (Confidential, Secret, Top Secret, etc.). Users cannot grant or change access to their own files; only the administrator can. Used in high-security environments — military, government, classified networks.

### DAC — Discretionary Access Control

The **data owner** (whoever created the resource) decides who else can read or modify it. Maximum flexibility, minimum central oversight. You create a spreadsheet, you decide which colleagues get read or edit. This is how Windows and most consumer OSes treat user-owned files by default.

> **Exam trap:** DAC = the **owner's discretion**. MAC = the **administrator's mandate**. The first word of the acronym tells you who holds the keys.

### RBAC — Role-Based Access Control

Permissions attach to **roles** (job functions): Manager, Director, Team Lead, Project Manager, Shipping Clerk. Users are added to roles; users inherit the role's permissions. The admin manages permissions on a small number of roles instead of a large number of users. In Windows this maps to **groups**.

### Rule-based access control

Permissions attached to **rules on objects**, evaluated at access time. Examples: "this file is accessible only between 9am and 5pm," "this form accepts submissions only from Chrome." Users don't set or own the rules — the administrator does. Rule-based is administrator-driven and **object-scoped**.

### ABAC — Attribute-Based Access Control

The newest model on the exam. Combines **multiple attributes** at evaluation time: requester IP, time of day, action (read vs write), user-to-data relationship, device posture, location. Lets administrators build complex policies — "allow write to this document only if the requester is in HR, on a corporate device, during business hours, from a US IP."

### Quick distinguishing table

| Model | Who decides | Granularity |
|---|---|---|
| **MAC** | Administrator, via labels | Coarse — by clearance level |
| **DAC** | Owner of each resource | Per-resource, per-user |
| **RBAC** | Administrator, via roles | Per-role |
| **Rule-based** | Administrator, via rules on objects | Per-object |
| **ABAC** | Administrator, via combined attributes | Per-request, multi-attribute |

### Time-of-day restrictions

A common modifier across most models. "R&D database accessible only 8am–6pm." "Conference-room network disabled after 8pm." A worldwide org has to handle time zones — the restriction is usually anchored to the user's locale, not the server's.

## Multifactor authentication (MFA)

Authentication factors are the *kinds* of evidence a user can present. MFA = two or more, from **different** categories.

| Factor | Examples |
|---|---|
| **Something you know** | Password, PIN, unlock pattern, security questions |
| **Something you have** | Smart card, USB security key, hardware token, software TOTP app, SMS code on your phone |
| **Something you are** | Fingerprint, voice print, face, iris (biometric) |
| **Somewhere you are** | GPS location, IP geolocation |

> **Exam trap:** Two passwords aren't MFA — same category. Password + phone-code is MFA. Password + fingerprint is MFA. Password + security question is *not* MFA — both are "something you know."

### Biometrics — what's actually stored

The system stores a **mathematical representation** (a template) of the biometric, not an image of your fingerprint or a recording of your voice. Hard to change if compromised — you can't issue yourself a new fingerprint — so biometrics are typically paired with another factor rather than used alone.

### Hardware vs software tokens

- **Hardware token** — physical fob displaying a rotating code; specific to the user.
- **Software token** — TOTP app on a phone (Google Authenticator, Authy). Phone itself counts as "something you have."
- **SMS code** — code texted to a known phone number. The weakest of the three (SIM-swap risk), but still better than no second factor.
- **USB security key** — certificate on the key; physical possession proves identity.

### Somewhere you are

GPS + IP geolocation can combine to estimate a user's physical location. Common use: deny logins from a country far from where the user was minutes ago. IPv6 makes pure IP geolocation harder, so GPS from a mobile device often complements it.

## Password security

### Entropy — why complexity rules exist

**Entropy** is how unpredictable a password is. Single dictionary words have low entropy and fall fast to **password spraying** and **brute-force** attacks. Mixed-case letters, digits, and special characters raise entropy per character. Length raises entropy faster than complexity — long passphrases beat short cryptic strings.

### Age, expiration, history

- **Password age** — how long the current password has been in use.
- **Expiration** — when the user must change it (commonly 30 / 60 / 90 days; critical systems may be 7 or 15).
- **Password history** — the system remembers previous passwords and rejects reuse.

Modern guidance is shifting away from frequent forced rotation toward longer passwords + MFA + breach monitoring, but the exam still tests the classic controls.

### Password managers

Encrypted local database storing per-site credentials. Master password (often plus MFA) unlocks the vault. Built into OSes (Keychain, Windows credential manager), available as third-party apps, and offered as Enterprise solutions. Most can generate strong random passwords, autofill, and flag reused or breached credentials.

### Passwordless authentication

Removes the user-typed password entirely. Authentication is via PIN (something you know), face/fingerprint (something you are), or hardware key (something you have). Windows Hello, mobile face/touch unlock, FIDO2 keys. Often paired with an initial password setup or a fallback factor — fully passwordless is a journey.

### Just-in-time (JIT) permissions

Used in environments where many technicians may need admin rights *briefly*. Instead of standing admin accounts:

```
Technician → request access from password vault
Vault → check policy → mint EPHEMERAL credentials based on primary credentials
Technician → does the work with temporary credentials
Session ends → ephemeral credentials are deleted
```

The vault holds the real **primary credentials**; they are never handed out. The technician sees only short-lived, scoped credentials. Limits blast radius if a tech's account is compromised — no standing admin rights to steal.

> **Exam trap:** JIT permissions ≠ MFA ≠ password manager. JIT specifically issues **ephemeral, scoped** credentials from a central vault on demand. The keyword is "temporary admin access for a specific task."

## How to read scenario questions

1. **Lifecycle phase?** Provisioning, change, or de-provisioning? Identity proofing happens *before* provisioning.
2. **Authentication or authorization?** "Logging in" → authentication (SAML, OpenID, password, MFA). "What can they do once logged in?" → authorization (OAuth, access-control models).
3. **Who decides access?** Owner (DAC), administrator with labels (MAC), administrator with roles (RBAC), administrator with rules on objects (rule-based), policy combining many attributes (ABAC).
4. **What KIND of factor?** Categorize before you count — two of the same category is still single-factor.
5. **Standing vs ephemeral access?** If the scenario mentions temporary admin rights from a central vault → JIT.
