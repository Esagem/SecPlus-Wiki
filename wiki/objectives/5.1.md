# 5.1 — Security Governance

CompTIA description: *"Summarize elements of effective security governance."*

Governance is the structure that turns intent ("be secure") into action. The exam tests three things: the **document hierarchy** (policies → standards → procedures), the surrounding **considerations** that shape it (legal, regulatory, industry, geography), and the **data roles** of the people responsible for what those documents protect.

---

## The document hierarchy

| Layer | Purpose | Specificity | Example |
|---|---|---|---|
| **Policy** | Statement of intent and rules | Broad | "We will protect customer data." |
| **Standard** | Measurable requirement | Specific | "Passwords must be ≥ 14 chars with one symbol." |
| **Procedure** | Step-by-step how-to | Granular | "To onboard: 1) sign AUP, 2) create AD account, 3) issue laptop…" |

Policies tell you *what and why*. Standards tell you *how much*. Procedures tell you *exactly how*. Scenario questions hinge on which rung the described control sits on.

---

## Security Policies

Documents that state what the organization requires for security and why. Broad in scope, mandatory in regulated environments, and the source from which standards and procedures derive. The master catalog is the **information security policy (ISP)** — every security policy in the org, plus the roles and responsibilities for who handles each scenario.

### Common policy types

| Policy | What it covers |
|---|---|
| **Acceptable use (AUP)** | What users may and may not do with company tech (laptops, phones, networks). Doubles as legal documentation if someone has to be dismissed for misuse. |
| **Business continuity (BCP)** | How to keep operating during a disruption. "Credit card network down → use paper slips and call-in approval." |
| **Disaster recovery (DRP)** | How to recover when a much larger event hits — fire, flood, ransomware. Recovery site, data recovery, application restoration, staff logistics. |
| **Incident response (IRP)** | What to do when a security event occurs — malware, DDoS, data exfiltration. Triggers IR roles. |
| **Software development lifecycle (SDLC)** | How the org builds software. Two common shapes: **Waterfall** (linear: requirements → dev → test → deploy → maintain) and **Agile** (iterative: design / develop / test / deploy / review per cycle). |
| **Change management** | How changes get proposed, reviewed, scheduled, executed, and rolled back. |

**BCP vs DRP:** BCP is *keep operating during disruption*; DRP is *recover when systems are down*. BCP is paper slips while the network's down. DRP is restoring the network itself. The exam pairs these often — same disaster, two phases.

**Incident response roles:** the **IR team**, **IT security management** (resourcing), **compliance officers** (mandate adherence), **technical staff** (hands on the problem), and the **user community** (eyes on the ground). NIST SP 800-61 r2 codifies the lifecycle: preparation → detection & analysis → containment, eradication & recovery → post-incident activity.

**Exam trap:** A policy isn't a control by itself — it's a *managerial* control (category) that's typically *directive* in type. Calling a policy "preventive" without context is wrong. The policy directs; the procedures and standards it spawns are what actually act.

---

## Security Standards

The measurable requirements that turn policy intent into something an auditor can check. Often sourced from external bodies like **ISO** (International Organization for Standardization) or **NIST** (National Institute of Standards and Technology), or written internally.

| Domain | Example standard |
|---|---|
| **Password** | Min length, complexity, rotation, manager allow-list, no local device accounts (must auth to AD via LDAP) |
| **Access control** | "Mandatory access control required; discretionary not permitted." Approval workflow, training prerequisites, deprovisioning triggers |
| **Physical security** | Badge required, biometric door locks in secure zones, visitor escort, motion detection in restricted areas |
| **Encryption** | Algorithms allowed (e.g., AES-256, SHA-256); salted hashes for password storage; separate rules for data at rest / in transit / in use |

**Standard vs Policy:** Policy says "passwords must be strong." Standard says "≥ 14 characters, one uppercase, one symbol, no dictionary words." The standard is what you can audit against.

**Exam trap:** ISO and NIST publish *standards*, not policies. Internal teams can also author standards — external sourcing is common but not required. Don't pick "ISO" as the answer to "who writes our policies" — that's the org's call.

## Security Procedures

The step-by-step playbooks that implement standards. Operational granularity — exactly what to do, in what order, when a specific event occurs.

### Change management (procedural form)
The lifecycle: **scope** the change → assess **risk** → write the **plan** → get **approval** → execute → **document**. Most orgs route every change through a **change control board (CCB)** that also requires a **backout plan**. Without a backout plan, a failed change can take the business down with no recovery path.

### Onboarding and offboarding

| Phase | Steps |
|---|---|
| **Onboarding** | Sign AUP / employee handbook → create accounts → assign rights and permissions → issue hardware |
| **Offboarding** | **Disable** account (don't delete — preserves decryption keys and data access) → recover hardware → reassign or archive data |

**Trap:** *Disable, don't delete.* Deleting an offboarded user can destroy access to encrypted files and stored data. Disabling preserves the cryptographic context while denying login.

### Playbooks and SOAR
A **playbook** is a procedure for a specific event — "recover from ransomware," "investigate data breach." When playbooks get automated and chained across third-party tools, they become **SOAR** (Security Orchestration, Automation, and Response), letting analysts focus on judgment calls instead of mundane steps.

### Governance shape

| Structure | Decision-making |
|---|---|
| **Board** | Sets broad objectives; delegates implementation to committees |
| **Committee** | Subject-matter experts who design how to meet objectives; report back to board |
| **Centralized** | One group decides for the whole org |
| **Decentralized** | Decisions made by the teams doing the work |

**Public vs private governance:** Public-sector governance carries legal and political constraints — meetings are often open to the public. Private-sector governance is more flexible and internally directed.

## Security Considerations

The external context that shapes which policies, standards, and procedures you need. Same data, different rules depending on who's regulating you and where you operate.

| Consideration | What it shapes |
|---|---|
| **Regulatory** | **SOX** (Sarbanes-Oxley, 2002) — financial reporting integrity; **HIPAA** — protected health information; **GLBA** — financial-institution privacy |
| **Legal** | Mandatory illegal-activity reporting; **legal hold** preserves data for litigation; breach disclosure laws vary by jurisdiction |
| **Industry** | Utilities / SCADA often air-gapped; healthcare emphasizes encryption + availability; finance emphasizes integrity |
| **Geographic** | Some countries require data on their citizens to **stay within national borders** (data sovereignty); GDPR for the EU |
| **Local / Regional / National / Global** | Wider scope → more concurrent legal regimes to comply with |

**Exam trap:** "Data must remain in-country" is *data sovereignty* — a geographic consideration, not a technical limitation. It's the answer when a question describes a cloud deployment blocked by jurisdiction.

## Data Roles and Responsibilities

Who is accountable for what about a given dataset. These roles also surface in privacy (5.4) and access control (4.6) questions.

| Role | Responsibility | Example |
|---|---|---|
| **Data owner** | Senior accountability for a class of data | VP of Sales owns customer relationship data; Treasurer owns financial data |
| **Data controller** | Decides **how** the data will be used | Payroll department |
| **Data processor** | Actually **processes** the data per controller's instructions | Third-party payroll company |
| **Data custodian / steward** | Day-to-day technical guardianship — security, accuracy, privacy, sensitivity labels, access control | DBA, records manager |

**Controller vs Processor:** Controller decides *what happens*; processor *does it*. Payroll dept (controller) tells the payroll company (processor) when and how to run paychecks. The processor never sets policy on its own.

**Owner vs Custodian:** Owner is *accountable* (executive-level, sets direction). Custodian is *responsible for the mechanics* (technical, day-to-day). One person can be both in a small org; in a large org they're separate.

**Exam trap:** Don't confuse data owner with system owner. Data owner is about the *information*. System owner is about the *platform that hosts it*. The exam usually wants data owner unless it specifically says "system."

---

## How to read scenario questions

For a 5.1 scenario, work in this order:

1. **What document type is described?** Statement of intent → policy. Measurable rule → standard. Step-by-step → procedure.
2. **What's the regulatory/industry/geographic context?** That sets which framework (SOX, HIPAA, GDPR, data sovereignty) is in play.
3. **Who is the actor?** Match them to data owner / controller / processor / custodian. The role decides what authority they have.
4. **If it's a change or an event,** check whether the question is asking about the *policy* (we must do X), the *standard* (X means at-least-Y), or the *procedure* (steps to do X).
