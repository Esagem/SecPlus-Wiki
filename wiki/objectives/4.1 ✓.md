# 4.1 — Securing Computing Resources

CompTIA description: *"Given a scenario, apply common security techniques to computing resources."*

The objective is wide: every kind of computing resource your enterprise owns — laptops, servers, mobile devices, wireless networks, ICS/SCADA, IoT, embedded — needs to be deployed in a known-good secure state and kept there. Five buckets cover it: **baselines**, **hardening targets**, **wireless/mobile**, **wireless security settings**, and **application security**.

---

## Secure baselines

A **secure baseline** is the agreed-upon set of security settings for an application instance — OS hardening, patch level, firewall rules, file permissions, and any app-specific config. It is the standard everything is measured against.

### The lifecycle

| Phase | What happens |
|---|---|
| **Create** | Build the baseline from vendor sources (app developer, OS manufacturer, appliance vendor) — don't start from scratch |
| **Deploy** | Push it out, often via Group Policy, MDM, or the Microsoft Security Compliance Toolkit (SCT) |
| **Maintain** | Audit existing instances against the baseline. Drift gets remediated. New vulnerabilities, OS changes, or app updates require baseline revision |

Windows 10 ships with **3,000+ Group Policy settings**. You don't tune them by hand — you start from a vendor template (Microsoft SCT, CIS Benchmarks) and adjust.

> **Exam trap:** Baselines and hardening guides come from different sources. The OS vendor gives a baseline; the app vendor gives another; appliance manufacturers give a third. Conflicts get resolved by the security team — pick the more restrictive one or document an exception.

---

## Hardening targets — the per-platform playbook

Every class of device has its own hardening list. Defaults are rarely secure; default credentials are never secure.

| Target | Key hardening moves |
|---|---|
| **Mobile devices** | Patch updates, segmentation (work vs personal data), MDM enrollment |
| **Workstations** (Windows / macOS / Linux) | Monthly patch cadence (Patch Tuesday), remove unused software, disable unused services |
| **Network infrastructure** (switches, routers, firewalls) | Change default credentials, centralized auth, vendor-only patches (rare but important) |
| **Cloud management workstations** | EDR installed, least privilege over cloud APIs, separate-cloud backups |
| **Servers** | OS patches / service packs, password length+complexity, least privilege, restrict communication to known peers, EDR |
| **SCADA / ICS** | Air-gapped network, very limited access, no internet exposure |
| **Embedded systems / RTOS** | Patches rare but high-priority when released, segment onto own network behind a firewall, run minimum services |
| **IoT** | High-priority patching (vendors aren't security companies), segment to own network |

### The recurring themes

- **Patch promptly** — but for SCADA/embedded/IoT, when patches are *available* they should be considered high-priority because they're released so rarely.
- **Default credentials are a vulnerability** — change them on switches, routers, firewalls, IoT, and anything else shipped with a known default.
- **Segment when patching isn't possible** — air-gapped network for SCADA, separate VLAN for IoT/embedded.
- **Least privilege everywhere** — especially on cloud management workstations, since they have broad cloud API access.

> **Exam trap:** SCADA / ICS questions almost always want "air gap" or "isolated network" as the answer. Real-time operating systems (RTOS) for industrial/military/automotive equipment add "deterministic timing" — they need isolation so other traffic doesn't disrupt timing.

## Securing wireless and mobile

### Site surveys and heat maps

Before deploying or troubleshooting wireless, inventory the RF environment:

| Tool | Purpose |
|---|---|
| **Site survey** | Identify existing APs (yours and neighbors), pick non-conflicting channels |
| **Heat map** | Visual signal-strength overlay on a floor plan |
| **Wireless survey tools** (e.g. NetSpot) | Enumerate SSIDs, BSSIDs, channels, bands |
| **Spectrum analyzer** | Find non-Wi-Fi interferers on the same frequencies |

### MDM and device-ownership models

**MDM** (Mobile Device Manager) is the management plane for fleet-wide mobile policy: push apps, enforce screen-lock policy, disable cameras inside HQ, partition company data from personal data.

| Model | Who owns the device | Notes |
|---|---|---|
| **BYOD** (Bring Your Own Device) | User | Hardest privacy/security balance; partition data; properly de-provision on phone trade-in |
| **COPE** (Corporate-Owned, Personally Enabled) | Company | Company assigns the device; personal use allowed |
| **CYOD** (Choose Your Own Device) | Company | Like COPE, but the employee picks the model |

### Mobile-connectivity threat surface

- **Cellular (4G/5G)** — traffic and location are partially visible to the carrier; not under your control.
- **Wi-Fi** — open networks (coffee shops, hotels) are unencrypted; needs VPN. Same-area attackers can do on-path attacks or RF interference (DoS).
- **Bluetooth** — short-range PAN; formal pairing is required, and auto-pairing is a risk. Don't pair with unknown devices.

## Wireless security settings

### WPA2 vs WPA3

| | WPA2 | WPA3 |
|---|---|---|
| Encryption | CCMP/AES | **GCMP** (Galois Counter Mode Protocol) |
| Integrity | CBC-MAC | **GMAC** (built into GCMP) |
| Handshake | 4-way handshake with PSK hash | **SAE** (Simultaneous Authentication of Equals) — Dragonfly |
| Brute-force exposure | Capture handshake, brute-force offline | Hash never crosses the air; offline brute-force defeated |
| Session keys | Shared from PSK | **Unique per client** even with the same PSK |

**SAE** is a Diffie-Hellman derivation that adds mutual authentication and per-session keys derived on each device. No 4-way handshake hash crosses the wire, so there's nothing to brute force.

> **Exam trap:** WPA2 is brute-forceable by capturing the 4-way handshake and cracking offline (GPU / cloud). WPA3 fixed this with SAE / Dragonfly. If the question mentions "captured the handshake and used a GPU rig," the answer is WPA2. If it mentions SAE, Dragonfly, or per-session keys with a shared PSK, the answer is WPA3.

### Authentication methods

| Mode | Use | Identifier |
|---|---|---|
| **Open** / None | Public Wi-Fi, no auth | Shows as "Open System" or "None" |
| **WPA3-Personal** (WPA-PSK) | Home, small office; everyone shares one passphrase | Pre-shared key |
| **WPA3-Enterprise** (WPA3-802.1X) | Corporate; per-user credentials | Username/password + auth server |

### 802.1X, EAP, AAA, RADIUS

| Component | Role |
|---|---|
| **802.1X** | Network Access Control (NAC) — gates access until auth succeeds. Wired and wireless. |
| **EAP** | Extensible Authentication Protocol — the framework that runs inside 802.1X, customizable per vendor |
| **Supplicant** | The client trying to connect (your laptop) |
| **Authenticator** | The first device you hit (AP, switch) — relays EAP, doesn't validate |
| **Authentication server** | The credential store (RADIUS) — the actual yes/no |
| **AAA** | Authentication, Authorization, Accounting — the framework RADIUS implements |
| **RADIUS** | Remote Authentication Dial-In User Service — the dominant AAA protocol |

The flow: supplicant → authenticator → authentication server → back. The authenticator never sees credentials in cleartext; it just relays.

> **Exam trap:** Don't confuse 802.1X with RADIUS. 802.1X is the *port-based* gating mechanism. RADIUS is the *backend AAA server*. EAP is the framework that rides on 802.1X to carry credentials to RADIUS.

> **Exam trap:** The three AAAs are **Authentication** (who are you — username+password), **Authorization** (what can you do — once you're in), **Accounting** (logged session metrics — time in, bytes, time out). Identification (the username alone) is separate from the three As.

## Application security

### Input validation and fuzzing

**Input validation** is the developer's job: every field accepting user input must be checked against an expected format (length, character set, range). Validation catches the kinds of bad input that drive injection attacks.

**Fuzzing** (dynamic testing) automates randomized input to see if the app responds unexpectedly — crashes, exceptions, unhandled errors. The first fuzzer came from a 1988 University of Wisconsin class project. Today's fuzzers can be platform- or language-specific.

### Cookies and secure cookies

Cookies are plain data — not executables, not malware — but their content can leak session state to anyone who reads them. **Secure cookies** have an attribute set that requires the cookie to travel only over HTTPS. **Never store sensitive data in a cookie** — the browser can be inspected at any time.

### Static code analysis (SAST)

**SAST** (Static Application Security Testing) reads source code looking for known-bad patterns: unsafe functions (`gets` instead of `fgets`), unbounded buffer operations, SQL string concatenation. Output is noisy — false positives are common, and the analyzer can't catch design-level issues like a bad crypto implementation. It's a filter, not a verdict.

### Code signing

The developer digitally signs the application binary with their private key, backed by a CA-issued certificate. At install time, the OS verifies the signature. Tells the user:
1. The code hasn't been modified since signing.
2. The signer's identity is verified by the CA.

If the signature is broken or absent, the OS prompts or refuses.

### Sandboxing

Restrict an application's environment so it sees only what it needs:

- **Development sandbox** — isolated build/test environment separated from production.
- **Runtime application sandbox** — the app at execution time has limited access (a VM is one form; mobile app sandboxes are another).
- **Mobile sandbox** — apps can't see each other's data without OS-mediated permissions; a compromised browser can't read your photos by default.

### Application monitoring

Built-in instrumentation during development lets ops see SQL injection attempts, abnormal file transfers, sudden access spikes — anything that hints at an attack or a previously unknown vulnerability.

## How to read scenario questions

1. **What's the resource?** Workstation, server, switch, mobile, SCADA, IoT, embedded, cloud admin host, app. Each has its own hardening playbook.
2. **What's missing from the secure state?** Patches? Default credentials? Segmentation? Encryption? MDM enrollment?
3. **For wireless:** PSK or 802.1X? WPA2 or WPA3? If a brute-force-captured handshake is in play, it's WPA2.
4. **For app code:** Is the problem at *write* time (SAST, code signing) or *runtime* (input validation, sandboxing, monitoring)?
