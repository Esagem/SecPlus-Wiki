# 3.3 — Data Protection

CompTIA description: *"Compare and contrast concepts and strategies to protect data."*

Three lenses for thinking about data: **what it is** (type and classification), **where it lives in its lifecycle** (state), and **how you defend it** (the seven protection techniques). The exam will give you a scenario and ask you to apply the right lens.

---

## The framework

| Lens | Question | Output |
|---|---|---|
| **Type** | What kind of data is this? | Regulated · PII · PHI · Proprietary · IP · Financial · Legal |
| **Classification** | How sensitive is it? | Public · Sensitive · Private · Confidential · Critical · Restricted |
| **State** | Where is it right now? | At rest · In transit · In use |
| **Sovereignty** | Where on earth is it stored? | Country-specific laws apply (GDPR, etc.) |
| **Protection** | How are we defending it? | Encryption · Hashing · Obfuscation · Masking · Tokenization · Segmentation · Permissions · Geofencing |

---

## Data types

| Type | What it covers | Notes |
|---|---|---|
| **Regulated** | Data a third party (regulator) dictates rules for | PCI cardholder data, HIPAA records, GDPR personal data |
| **Trade secret** | Internal-only proprietary process or formula | Protected by secrecy itself; no public disclosure |
| **Intellectual property (IP)** | Creative or inventive work | Protected by copyright, trademark, patent — often visible but legally restricted |
| **Legal information** | Court records, attorney-client material | Public/private split — some records public, some sealed |
| **Financial** | Corporate financials, payments, account info | Internal financials private; consumer financial data is regulated |
| **PII** (Personally Identifiable Information) | Anything that points back to a specific person | Name, DOB, SSN, mother's maiden name, biometrics, address |
| **PHI** (Protected Health Information) | Health-specific data tied to a person | Records, diagnoses, payments for care — HIPAA-regulated in the US |
| **Proprietary** | Owned and unique to the organization | Overlaps with trade secret and IP |

Also useful: **human-readable** (a doc, a spreadsheet) vs **non-human-readable** (barcode, encoded blob, hashed value).

> **Exam trap:** PII is identity-linking; PHI is the medical subset. All PHI that contains identity is also PII, but plain PII (just a name and address) isn't PHI.

---

## Classification levels

A spectrum from open to locked-down. Exact labels vary by org and country, but the relative order is what the exam tests.

| Level | Access | Example |
|---|---|---|
| **Public / Unclassified** | Anyone | Marketing pages, published quarterly reports |
| **Sensitive** | Limited internal | IP, generic PII |
| **Private** | Need-to-know with permissions | Employee records |
| **Confidential** | Tighter need-to-know, often NDA-gated | Strategic plans, M&A docs |
| **Restricted / Classified** | Strict permissions, may require NDA or clearance | Government classified levels |
| **Critical** | Must always be available | Business-continuity-required data |

> **Exam trap:** "Critical" in classification language is about *availability*, not just sensitivity — losing access matters as much as preventing it.

## States of data

Where data sits determines what protection works. The three states:

| State | Where | How to protect |
|---|---|---|
| **At rest** | On a storage device — disk, SSD, USB, database | Full disk encryption, file/folder encryption, DB column encryption, ACLs |
| **In transit** (in motion) | Crossing a network — LAN, WAN, internet | TLS, IPsec, VPN; firewalls/IPS for traffic policy |
| **In use** | In RAM or being processed by the CPU | Almost always decrypted here — protect the host: memory protection, EDR, hardened endpoints |

### The Target lesson

The 2013 Target breach lifted ~110 million credit card numbers by scraping point-of-sale **memory** — the cards were encrypted at rest and in transit, but had to be decrypted in RAM to be processed. Attackers go for data in use precisely because that's where encryption usually isn't.

> **Exam trap:** If the scenario says "encrypted at rest and in transit but still breached," the attack is almost certainly against data in use.

### Data sovereignty

Data stored in a country is subject to that country's laws — court orders, retention rules, disclosure rules. **GDPR** (EU) requires data on EU citizens to be stored in the EU. Plan storage location to match the laws you can comply with.

## The seven protection techniques

### Encryption

Transforms plaintext into ciphertext so unauthorized readers see nothing useful. Reversible with the key. Property to remember: **confusion** — small changes in plaintext produce dramatic changes in ciphertext. Apply to data at rest (FDE, file/folder, column-level) and data in transit (TLS, IPsec).

### Hashing

One-way fingerprint. Same input → same hash; different inputs should produce *very* different hashes. Cannot be reversed. Uses:

- Password storage (store hash, compare hashes at login)
- File integrity (download a file, hash it, compare to published hash)
- Digital signatures (sign the hash, not the whole file)

A **collision** is when two different inputs produce the same hash output. Algorithms with known collision weaknesses (MD5, SHA-1) get retired in favor of stronger ones (SHA-256, SHA-3).

> **Exam trap:** Hashing is **not** encryption — there's no key and no way to recover the original. "I need to compare passwords but never retrieve them" → hashing. "I need to send a message that the recipient can read" → encryption.

### Obfuscation

Make the data hard for a human to read without changing what it does. Code obfuscation hides logic from someone reading the source; both the original and obfuscated code execute identically. Attackers also use it to hide malicious scripts — which is why deobfuscation tools exist.

### Masking

Show *part* of the data, hide the rest. Receipts that show only the last four digits of a credit card (`****-****-****-1234`). The full data still exists in the backend; the user-facing view is masked.

### Tokenization

Replace sensitive data with a meaningless **token**; the real data never leaves a secure vault. Common on mobile payments — your phone sends a one-time token, not your card number. Even if intercepted, the token can't be replayed. No encryption math involved; the protection is in the indirection.

> **Exam trap:** Tokenization vs encryption. Encryption is reversible math on the actual data. Tokenization swaps the data for a stand-in that has no math relationship to the original. If the question stresses "no encryption keys," "one-time," or "can't be reversed even with full system access," it's tokenization.

### Segmentation

Split a monolithic data store into smaller, separately-secured pieces. The Target-class breach is much harder when there's no single database that holds everything. Apply differentiated security per segment based on what's inside.

### Permission restrictions

The everyday baseline. Strong authentication + per-account rights + group memberships + file/object ACLs. The login is only the gate; post-login permissions decide what the user can actually touch.

### Geographic restrictions and geofencing

Use location as an access factor. Mechanisms:

- **IP subnet** — accurate for fixed corporate networks
- **GPS** — accurate for mobile devices
- **Wireless SSID databases** — cross-reference visible Wi-Fi to known locations

**Geofencing** = "only allow access inside this geographic boundary" (e.g., only on the corporate campus, only inside the country). Geographic restriction is the broader policy; geofencing is the specific implementation.

## How to read scenario questions

1. **Name the data.** PII, PHI, regulated, proprietary? That sets which laws and standards apply.
2. **Name the state.** At rest, in transit, in use? That tells you which protection mechanism fits.
3. **Pick the technique by goal:**
   - Need it back, in clear, with a key → **encryption**
   - One-way fingerprint → **hashing**
   - Hide some of it from human view → **masking**
   - Swap for a stand-in token → **tokenization**
   - Make code/data unreadable but functional → **obfuscation**
   - Reduce blast radius → **segmentation**
   - Gate by identity → **permissions**
   - Gate by location → **geofencing**
