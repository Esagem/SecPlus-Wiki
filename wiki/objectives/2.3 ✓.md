# 2.3 — Vulnerabilities

CompTIA description: *"Explain various types of vulnerabilities."*

Where 2.2 was the *path* (vector) and 2.4 will be the *action* (attack), this objective is the *weakness* — the underlying flaw that makes an attack possible. The exam asks you to classify weaknesses by where they live (application, OS, hardware, virtualization, cloud, supply chain, mobile, configuration) and to recognize the canonical mechanism behind each.

---

## The vulnerability map

| Family | Where the weakness lives | Headliners |
|---|---|---|
| **Application** | Software written by your team or a vendor | Memory injection, buffer overflow, race condition, SQL injection, XSS |
| **Operating system** | The OS itself | Unpatched kernel/userland bugs |
| **Hardware / firmware** | The silicon and the firmware that runs it | EOL/EOSL devices, legacy systems |
| **Virtualization** | The hypervisor and VM boundary | VM escape, resource reuse |
| **Cloud** | The shared-responsibility line | Unpatched cloud apps, MFA gaps, misconfig |
| **Supply chain** | Code, hardware, and services from third parties | Malicious updates (SolarWinds), counterfeit hardware |
| **Misconfiguration** | Anything operated incorrectly | Open buckets, weak admin, insecure protocols, default creds, open ports |
| **Mobile** | Phones and tablets | Jailbreaking, rooting, sideloading |
| **Zero-day** | Anywhere — defined by *who knows about it* | Vulnerability with no patch yet |

Zero-day is orthogonal to the rest — it's a *status*, not a location. Any of the families above can have a zero-day variant.

## Application vulnerabilities

### Memory injection
Malware running inside an existing process's memory rather than as its own process. The classic example is **DLL injection**: the attacker places a malicious DLL on disk and forces a legitimate process to load it. Once loaded, the malware inherits the host process's rights and permissions — often a fast path to privilege escalation.

**Exam trap:** memory injection helps malware avoid detection because the malicious code runs *inside* a trusted process. Anti-malware looking for unfamiliar processes won't see it.

### Buffer overflow
Writing more data into a memory region than that region is sized to hold, spilling into adjacent memory. If the spill changes a variable that controls security (a privilege flag, a return address), the attacker may be able to escalate privileges or redirect execution.

- **Found through:** fuzzing and code review for missing bounds checks.
- **Hard to exploit reliably:** often crashes the application instead of yielding a working exploit. What makes a buffer overflow valuable is **repeatability** — the same input always produces the same useful effect.
- **Mitigations:** bounds-checking by the developer, DEP (data execution prevention), ASLR (address space layout randomization).

### Race condition / TOCTOU
A flaw where two events happen close enough in time that the application doesn't handle the simultaneity correctly. The named subtype is **TOCTOU** — Time-Of-Check to Time-Of-Use — where the application checks a value, then later uses it, and an attacker changes the value between the two steps.

The Mars Rover Spirit hit a reboot-loop race condition in 2004. The 2023 Pwn2Own Tesla Model 3 root exploit was a TOCTOU against the infotainment Bluetooth stack. Bank-transfer races where deposits update immediately but withdrawals don't update until later is the canonical exam example.

**Exam trap:** TOCTOU is a *kind* of race condition. If the stem says "checks then uses with another process in between," it's TOCTOU even if the question never names it.

### SQL injection (SQLi)
The attacker inserts SQL syntax into an input field that the application later concatenates into a database query. If the application doesn't sanitize or parameterize, the database executes the attacker's SQL alongside the legitimate query.

```
SELECT * FROM users WHERE name='Professor' OR '1'='1';
```

`OR '1'='1'` is the signature: it makes the WHERE clause always true, returning everything. SQLi can read, modify, delete, or destroy data, or in some cases pivot to the underlying OS.

**Primary defense:** parameterized queries / prepared statements. Input validation is secondary.

### Cross-site scripting (XSS)
Abbreviated **XSS** (not CSS — that's taken by stylesheets). The attacker gets the victim's browser to execute attacker-supplied script in the context of a trusted site, usually via inadequate input sanitization on the site.

| Type | How it propagates |
|---|---|
| **Non-persistent (reflected)** | Malicious script is in a crafted URL/link sent to the victim. Browser sends it to the vulnerable site, which reflects it back. Runs once for the recipient. |
| **Persistent (stored)** | Malicious script is stored on the server (forum post, profile field, comment) and runs in every visitor's browser. |

What XSS actually steals: cookies, session IDs, CSRF tokens, anything the victim's authenticated session can access. The 2017 Subaru bug paired XSS with a non-expiring auth token to enable full remote control of customers' vehicles.

**Defense:** input validation everywhere user data enters, output encoding everywhere user data leaves, content security policy, keep browser and app patched.

## Operating system vulnerabilities
The OS is huge (tens of millions of LOC in modern Windows) and patched continuously. Microsoft's **Patch Tuesday** — the second Tuesday of each month — is the named monthly cycle the exam expects you to know. A typical Patch Tuesday rolls 50–100 fixes covering elevation of privilege, RCE, security-feature bypass, and information disclosure bugs.

Practices to memorize:
- Test patches in a staging environment before production where possible.
- Maintain backups before applying patches.
- Some patches require a reboot to take effect; plan around it.
- Track vulnerabilities at **cve.mitre.org** (Common Vulnerabilities and Exposures) and **msrc.microsoft.com** for Microsoft specifically.

---

## Hardware vulnerabilities

The exam's focus is what to do when you *can't* patch.

| Term | Meaning |
|---|---|
| **Firmware** | The OS embedded in a hardware device (router, thermostat, IoT). Only the manufacturer can typically patch it. |
| **EOL (End of Life)** | Manufacturer has stopped selling the product. Support and patches usually still available for some window. |
| **EOSL (End of Service Life)** | Manufacturer has stopped providing patches. The device is now a permanent vulnerability unless removed or compensated. |
| **Legacy** | An older system still in use because something critical depends on it. Often EOL or EOSL. |

Real-world cadence: Trane Comfortlink II thermostats were reported vulnerable in April 2014 and not patched until April 2015 — a year. IoT firmware cycles are nothing like Microsoft's monthly rhythm.

**For legacy/EOSL systems that can't be retired:** apply compensating controls — firewall rules, IPS signatures tailored to the legacy OS, network segmentation.

## Virtualization vulnerabilities

### VM escape
A flaw that lets code inside one VM break out and reach the hypervisor or other VMs on the same host. Catastrophic when it works — one compromised guest can pivot to every other guest. The 2017 Pwn2Own demonstration chained an Edge JS bug → Edge sandbox → Windows 10 kernel → VMware hardware-simulation bug to escape the VM.

### Resource reuse
The hypervisor over-allocates physical resources (especially RAM) across VMs assuming they won't all use it simultaneously. If hypervisor bugs allow one VM to read memory that was written by another VM, sensitive data can leak across the VM boundary.

**Both are mitigated by:** keeping the hypervisor patched, isolating VMs by sensitivity, monitoring hypervisor advisories.

## Cloud-specific vulnerabilities

The shared-responsibility model means many vulnerabilities are configuration, not platform. Exam-frequent items:

- **Public exposure** — apps in a public cloud are reachable from anywhere in the world. Anyone can attempt to authenticate, fuzz the API, or DDoS.
- **Weak/misconfigured authentication** — surveys cited in the video find 76% of orgs don't enable MFA for the cloud management console. That's the single most catastrophic misconfiguration possible.
- **Unpatched cloud code** — up to 63% with CVSS ≥ 7 unpatched.
- **Directory traversal** — web server misconfig lets the attacker move into folders outside the web root.
- **Remote code execution via app vulns** — Log4j and Spring Cloud Function are the cited examples.
- **Out-of-bounds writes**, **SQLi** against cloud-hosted databases, **XSS** against cloud-hosted web apps — same families as application vulnerabilities, just hosted in someone else's data center.

## Supply-chain vulnerabilities

Three layers, all on the exam:

1. **Hardware** — counterfeit or implanted devices (counterfeit Cisco resold for ~$1B before DHS arrested the reseller in 2022).
2. **Services** — managed service providers, MSPs, and other vendors with privileged access. Target's HVAC contractor (2013) is the canonical example.
3. **Software** — malicious updates distributed through trusted channels.

### Malicious updates and the SolarWinds case
**SolarWinds Orion** (March/June 2020) is the case study to memorize. Attackers got into SolarWinds' build pipeline, injected malicious code into Orion, and the malicious build was **digitally signed** by SolarWinds and distributed through Orion's automatic update mechanism. ~18,000 customers received it, including Microsoft, Cisco, Intel, the Pentagon, DHS, and Treasury. Not detected until December 2020.

The lesson is *trust is conditional*: a signed update from a legitimate vendor is still only as trustworthy as the vendor's build pipeline.

**Best practices for any update:**
- Back up before updating.
- Use only trusted sources.
- Verify digital signatures.
- Monitor for unusual behavior even after a "successful" update.

## Misconfiguration vulnerabilities

The single most common cause of breaches in practice. Exam recurring examples:

- **Open cloud storage** — S3 buckets without authentication (Verizon 14M records, 2017).
- **Weak/unsecured admin accounts** — root/Administrator with passwords like `ninja` or `football`. Best practice: disable direct admin login; require elevation (`sudo`, "Run as administrator") from a normal account.
- **Insecure protocols** — Telnet, FTP, SMTP, IMAP, HTTP send credentials and data in cleartext. Replace with SSH, SFTP, https, IMAPS, etc. DefCon's "Wall of Sheep" exists to publicize this.
- **Default credentials** — Mirai botnet again.
- **Open ports / firewall sprawl** — over-permissive rules. Mitigation: regular firewall audits, principle of least exposure.

## Mobile device vulnerabilities

| Term | Platform | What it does |
|---|---|---|
| **Jailbreaking** | iOS | Replaces or modifies firmware to bypass Apple's restrictions |
| **Rooting** | Android | Replaces or modifies firmware to gain root/superuser access |
| **Sideloading** | Both | Installing apps outside the official app store(s) |

All three circumvent the security model that the platform vendor and your MDM (Mobile Device Manager) rely on. A jailbroken device can sideload anything, evade MDM policy, and run unsigned code — every MDM control assumes the OS is intact.

**Exam framing:** jailbreaking/rooting is generally forbidden in employee AUP and is typically grounds for dismissal in enterprise environments.

---

## Zero-day vulnerabilities

A **zero-day** is a vulnerability for which **no patch yet exists** because the vendor is unaware (or the patch hasn't shipped). The attacker discovered it first; defenders have "zero days" to prepare.

Zero-day attacks bypass signature-based defenses by definition — no signature exists for an unknown attack. Defense is layered: behavior-based detection, EDR, network segmentation, least privilege — anything that catches *what the attack does* rather than *what it looks like*.

Tracking: **cve.mitre.org** publishes CVEs as they're disclosed and patched.

**Exam trap:** zero-day isn't a vulnerability *type* — it's a vulnerability *status*. A SQL injection bug can be zero-day for a few hours after the attacker discovers it; the same bug stops being zero-day the moment the vendor knows and ships a patch.

---

## How to read scenario questions

1. **Where does the weakness live?** Application code? OS? Firmware? Hypervisor? Cloud config? Third party? Mobile OS? That puts you in the right family.
2. **What is the precise mechanism?** Overflow, race, injection, escape, traversal. The name almost always maps directly to a vocab term.
3. **Is the question about the vulnerability or the patch state?** "No patch is available" → zero-day. "Patch exists but isn't applied" → unpatched system, not zero-day.
4. **For misconfiguration questions:** the answer is rarely "the platform is buggy." It's almost always "the configuration was wrong" — defaults, weak admin, insecure protocol, open port.
