# 2.5 — Mitigation Techniques for Security

CompTIA description: *"Explain mitigation techniques used to secure the enterprise."*

This objective is the defensive counterweight to 2.4. Where 2.4 names the attacks, 2.5 names the controls that stop them. Most of the answers are layered: segmentation contains blast radius, ACLs gate traffic, hardening shrinks attack surface, and EDR catches what slipped through.

---

## The mitigation stack

| Layer | Techniques |
|---|---|
| **Network shape** | Segmentation (physical, logical/VLAN, virtual), ACLs |
| **What can run** | Application allow lists, deny lists |
| **Keep current** | Patching, posture assessment |
| **Protect the data** | File-level encryption, FDE, application encryption |
| **See what's happening** | Monitoring, logging, SIEM |
| **Limit blast radius** | Least privilege, host firewall, HIPS, EDR |
| **Shrink attack surface** | Hardening, close ports, change defaults, remove unused apps, decommission cleanly |

---

## Segmentation

Splitting the network into smaller pieces so that a security event in one piece can't reach the others.

| Type | Mechanism |
|---|---|
| **Physical segmentation** | Separate physical switches, separate cables — no shared infrastructure |
| **Logical segmentation** | Same physical switch, multiple **VLANs** — traffic separated in software |
| **Virtual segmentation** | Cloud / VM architectures — segmented inside the hypervisor or cloud fabric |

Reasons to segment:

- **Performance** — dedicate bandwidth to a high-throughput app so other users don't degrade it.
- **Security** — users talk to the app tier; only the app tier talks to the database. A compromised user workstation can't reach the DB directly.
- **Compliance** — PCI requires credit-card data to be isolated from the rest of the network.

**Exam trap:** "VLAN" alone is **logical** segmentation, not physical. Two devices on the same switch in different VLANs share the wire — the separation is in software.

---

## Access Control Lists (ACLs)

Rules on a network device, OS, or application that allow or deny traffic based on a combination of attributes. The named grouping attributes:

- Source IP address
- Destination IP address
- Port numbers
- Time of day
- User / group identity

Example rule shape: *"James can access 192.168.1.0/24 on TCP ports 80, 443, and 8088."* The more specific the rule, the more granular the control.

ACLs also live in the OS — file/folder permissions and group memberships are ACLs by another name.

**Exam trap:** When building ACLs, don't lock yourself out. A rule that denies all traffic to the management interface from your own subnet will kill your next session.

---

## Application allow lists vs deny lists

Two opposing philosophies for what's permitted to execute:

| | Allow list | Deny list |
|---|---|---|
| **Default behavior** | Block everything | Allow everything |
| **List contains** | Only the apps that may run | The apps that may NOT run |
| **Posture** | Restrictive — secure but high-friction | Permissive — flexible but reactive |
| **Familiar example** | Locked-down kiosk / regulated workstation | Anti-malware (blocks known bad, lets everything else run) |

Windows identifies apps in allow/deny rules by:

- **Hash** — exact binary fingerprint; any change invalidates the rule.
- **Certificate** — digital signature; trust an entire publisher (Microsoft, Adobe).
- **File path** — only allow execution from specific directories.
- **Network zone** — different rules depending on whether the system is on a private or public network.

**Exam trap:** AV is a **deny list** (signatures of known bad). A scenario describing "only approved apps can run, everything else is blocked by default" is an **allow list** — the more restrictive of the two.

---

## Patching

The single highest-leverage mitigation. Catalog the patterns:

- **Automatic patching** — common on home OS installs; the OS checks, downloads, installs without user action.
- **Tested deployment** — enterprise pattern. IT validates the patch in a test environment before pushing to production.
- **Emergency / out-of-band patches** — released outside the normal cycle when attackers are actively exploiting the vulnerability.

Patching covers OS, applications, and device firmware. Missing one layer leaves the gap open even if the others are current.

---

## Encryption

Three layers — pick the right one for the threat:

| Layer | Mechanism | Use case |
|---|---|---|
| **File-level** | OS encrypts selected files/folders. **Windows EFS** is the canonical example. | Protecting specific sensitive files on a shared system |
| **Full Disk Encryption (FDE)** | Encrypts the whole volume — OS, apps, user data. **BitLocker** (Windows), **FileVault** (macOS). | Laptops and any device that leaves the building |
| **Application-level** | The app encrypts its own data, independent of file-system or disk encryption. | Apps that can't assume FDE is in place, or that need encryption end-to-end |

**Exam trap:** FDE protects against an offline attack (drive removed and read in another system). FDE does **not** protect a running, unlocked system — once the user authenticates, the data is decrypted in memory.

---

## Monitoring, logging, SIEM

Visibility is a mitigation — you can't respond to what you can't see.

- **Built-in monitoring** lives in switches, routers, firewalls, IPS, and authentication systems. Each of those produces logs.
- **External sensors** sit outside those devices and observe traffic independently.
- **SIEM (Security Information and Event Manager)** consolidates logs from all sources into one place for correlation, reporting, and alerting.

Without a SIEM, logs are scattered across dozens of systems. With it, the analyst can correlate events across the whole environment.

---

## Least privilege

A user gets the minimum rights needed to do their job — no more. Best case: no user runs day-to-day with administrative permissions. If a task needs admin rights, they elevate temporarily, then drop back to normal.

The payoff is at breach time. Malware inherits the privileges of the account it lands in. A user-level account limits the attacker to user-level data; an admin account hands over the whole system.

---

## Posture assessment

Performed at login or network connection. The system being connected is checked against organizational policy:

- OS version and patch level
- AV / EDR version and signature freshness
- Local firewall and EDR configuration
- Certificate proving the device is owned by the org

A system that fails the check is dropped into a **quarantine VLAN** — limited connectivity, just enough to remediate. After updates are installed, the system re-authenticates, passes the check, and joins the production network.

**Exam trap:** Posture assessment is about the *connecting endpoint's configuration*, not the user's credentials. A user can pass authentication and still fail the posture check.

---

## Decommissioning

Equipment that's leaving service may still hold sensitive data on its storage.

- **Re-use within the organization** — format the drive and move it to another internal system.
- **Drive removal** — pull the storage drive before disposing of the chassis.
- **Destruction** — physical destruction (shredding, degaussing, pulverizing) for drives containing data that must never be recovered.

Just deleting files isn't enough. Just formatting isn't enough against forensic recovery. For high-sensitivity data, destruction is the only safe answer.

---

## Hardening (system level)

The set of baseline configurations that make a system harder to attack:

| Technique | What it does |
|---|---|
| **Apply security patches** | Monthly cycles plus emergency patches for active exploits |
| **Password policy** | Minimum length + complexity (upper, lower, number, special) |
| **Limited account access** | Not every account is an admin; users only have the rights their job requires |
| **IP-range restrictions** | Only specific source IPs may reach the management interface |
| **AV / endpoint protection** | Continuous monitoring with current signatures |
| **Encrypt data at rest and in transit** | File-level / FDE for storage; VPN / HTTPS for the wire |

---

## Endpoint hardening

The same logic applied to each user device — desktop, laptop, tablet, phone. The OS, apps, and threats differ per platform, so the controls do too. No single switch flips on security; you stack tools for **defense in depth**.

---

## EDR (Endpoint Detection and Response)

The next-generation answer to "over a million new malware variants per day." Traditional AV depends on signatures and can't keep up.

What EDR adds beyond signatures:

- **Behavioral analysis** — flags malicious patterns even when no signature exists.
- **Machine learning** — rapid classification of new malware families.
- **Process monitoring** — every new process is watched in real time.
- **Root-cause analysis** — once something suspicious fires, EDR investigates upstream to find the entry point.
- **Automated response** — isolate the system, quarantine the file, roll back configuration — without waiting for a ticket.
- **API-driven** — central management console, fleet-wide visibility and control.

**Exam trap:** "Signatures only" is **traditional AV**. The moment a question mentions behavior, process monitoring, ML, automated isolation, or root-cause analysis — the answer is **EDR**.

---

## Host-based firewall

Software firewall on the OS itself. Two properties that matter for the exam:

- **Sees traffic before/after encryption** — because it runs on the endpoint, it can inspect data on either side of the crypto boundary.
- **Manages inbound *and* outbound** — outbound matters because malware on the host needs to talk to its C2 server.

Manageable from a central console even though each instance runs locally.

---

## Host-based IPS (HIPS)

The IPS pattern (signatures + heuristics + behavioral) applied at the OS. Often bundled with EDR or anti-malware.

HIPS sees what a network IPS can't:

- **Buffer overflow** in a running process
- **Registry changes**
- **Modifications to core OS files**

It can alert and block in-process.

**Exam trap:** A *network* IPS only sees traffic. A *host-based* IPS sees the OS internals — registry, files, process memory. Choose by what the scenario observes.

---

## Open ports and services

Each open port is an attack surface. Mitigations:

- Close ports for services you don't run.
- Use a firewall — ideally a **next-generation firewall** that filters by application, not just by port number.
- Audit what's actually listening with **nmap**.
- Refuse software that demands wide-open port ranges (the famous antipattern: "open ports 0–65535"). That's the vendor avoiding support calls, not a real requirement.

---

## Default credentials

Every router, switch, firewall, access point, and many applications ship with a default admin login. The defaults are publicly documented — attackers scan for them.

Mitigations:

- Change the default at first login (some devices force it, many don't).
- Add **MFA** to the management interface.
- Centralize authentication so device logins flow through a directory you control, not a local admin account.

---

## Remove unused applications

Every installed app has its own patching process and its own vulnerabilities. The simplest answer to a vulnerable app you don't use: **uninstall it**. One less thing to patch, one less attack surface.

---

## How to read scenario questions

1. **Identify the layer the scenario lives at.** Network shape → segmentation/ACL. What runs → allow/deny lists. Endpoint → hardening, EDR, HIPS, host firewall.
2. **Distinguish similar pairs:**
   - File-level encryption vs FDE → specific files vs entire volume.
   - Allow list vs deny list → restrictive default-deny vs permissive default-allow.
   - Traditional AV vs EDR → signatures only vs behavior + ML + automated response.
   - Network IPS vs HIPS → traffic on the wire vs OS internals.
   - Physical vs logical segmentation → separate switches vs VLANs on shared switches.
3. **"While waiting for…" or "the official fix isn't ready" usually means a compensating control** (covered in 1.1) — e.g., a firewall rule standing in for a missing patch.
4. **Watch for posture vs authentication.** "Logged in but quarantined" is a posture-assessment outcome, not an auth failure.
