# 3.1 — Architecture and Infrastructure Models

CompTIA description: *"Compare and contrast security implications of different architecture models."*

This objective is a survey of *where* and *how* compute and data live — cloud vs on-prem, virtualized vs containerized, monolithic vs microservices, IoT/SCADA/RTOS/embedded, and the cross-cutting considerations (availability, cost, responsiveness, scalability, patchability, power). The exam doesn't ask you to design these; it asks you to recognize the model and name its security trade-off.

---

## The framework

Two questions to ask of every architecture:

1. **Where does the workload run?** — on-prem · public cloud · hybrid/multi-cloud · IoT/SCADA/embedded.
2. **How is the workload packaged?** — monolithic · virtualized · containerized · serverless · microservices.

Pick a cell and you've located the security model. The implications then come from who owns what (responsibility matrix), what's exposed (attack surface), and what fails gracefully (resilience).

## Cloud infrastructures

### Responsibility matrix

Every public cloud provider publishes one — a grid of who patches, configures, and secures each layer across **IaaS · PaaS · SaaS · on-prem**. The customer's slice shrinks as you move left-to-right (more managed by provider), but **accounts and identities are always the customer's** regardless of service model.

| Layer | On-prem | IaaS | PaaS | SaaS |
|---|---|---|---|---|
| Data & access | Customer | Customer | Customer | Customer |
| Application | Customer | Customer | Customer | Provider |
| OS | Customer | Customer | Provider | Provider |
| Virtualization | Customer | Provider | Provider | Provider |
| Hardware / network | Customer | Provider | Provider | Provider |

> **Exam trap:** "Who is responsible for patching the OS in PaaS?" → Provider. "Who is responsible for the user accounts in SaaS?" → Customer. The line doesn't move just because the service model moved.

### Hybrid and multi-cloud

Two or more clouds (often plus on-prem). Flexibility, but configuration drift is the killer — auth, firewall rules, log formats, and APIs differ across providers, and data crossing providers traverses the public internet. Each crossing is a transit-encryption decision.

### Third-party vendors

Cloud rarely arrives alone. A third-party WAF in front of your app, a third-party SIEM, a third-party identity provider — each one is a separate vendor risk surface. **Vendor risk management policy** + extending IR runbooks to include third parties are the controls the exam expects.

### Infrastructure as code (IaC)

The architecture is described as a text file (Terraform, CloudFormation, Bicep) and applied to build identical environments on demand. Benefits: reproducibility, version control, fast rebuild. Security concern: a misconfiguration committed once propagates to every environment built from that code.

### Serverless

No server instances — the cloud runs individual **functions** on demand. Customer never sees the OS. Security shifts almost entirely to the cloud and to function-level auth. Cost-efficient because functions only run when invoked.

### Microservices vs monolithic

| | Monolithic | Microservices |
|---|---|---|
| Shape | One big executable | Many small services behind an API gateway |
| Update | Replace whole app | Replace one service |
| Scale | Scale the whole app | Scale only the hot service |
| Failure mode | Whole app down | Other services keep working |
| Security | One policy fits all | Per-service policy |

> **Exam trap:** "API gateway" is the dead giveaway for microservices. Monolithic apps don't have one.

## Network infrastructure concepts

### Air gap vs VLAN — physical vs logical isolation

| | Air gap | VLAN |
|---|---|---|
| Mechanism | Two separate physical switches, no cable between | One switch, ports tagged into separate broadcast domains |
| Cost at scale | Linear — 100 tenants = 100 switches | Flat — one switch handles many VLANs |
| Bypass risk | Requires physical access | Misconfigured trunk or VLAN-hopping attack |
| Use case | Highly classified networks, MSP per-tenant separation | Standard enterprise segmentation |

> **Exam trap:** "Physically separated, no connectivity" → air gap. "Logically separated on the same switch" → VLAN. Both are valid segmentation; the question hinges on whether the cable exists.

### SDN — three planes

Software-defined networking pulls apart what a switch or router does into three layers that can be coded as software:

| Plane | What it does | Example |
|---|---|---|
| **Data plane** | Forwards packets, runs NAT, encrypts trunks — the actual "moving traffic" work | Switch fabric, line cards |
| **Control plane** | Decides *how* traffic should move — routing tables, ARP tables, session state | OSPF, BGP, dynamic routing |
| **Management plane** | Configures the device — SSH, SNMP, API calls | Admin SSH session, NetConf, vendor API |

Pulling these apart in software is why you can spin up a "firewall" or "router" in the cloud with a click — the physical box has become three software components you can recompose at will.

> **Exam trap:** Configuring the device = management plane. Choosing the next hop = control plane. Actually moving the packet = data plane. Don't confuse "control" with "management."

## Other infrastructure concepts

### On-premises vs cloud

| | On-prem | Cloud |
|---|---|---|
| Security location | Local, you own the stack | Provider-managed, centralized with services |
| Control | Complete | Constrained by what provider exposes |
| Staff cost | High (skilled in-house team) | Lower (consume as service) |
| Capex / opex | Capital expense, depreciates | Operating expense |
| Change speed | Slow (procure, install) | Fast (click) |

### Centralized vs decentralized

A centralized console (SIEM, single management plane) gives one place to monitor everything — great for visibility, terrible as a single point of failure. As the org grows, the console itself becomes a scaling problem.

### Virtualization vs containerization

| Layer (top → bottom) | Virtualization | Containerization |
|---|---|---|
| App | Application | Application |
| OS | **Guest OS per VM** | (none — shared) |
| Runtime | Hypervisor | Container engine (e.g., Docker) |
| OS | (hypervisor type 1: none) | **Host OS, shared by all containers** |
| Hardware | Infrastructure | Infrastructure |

Containers are lighter — no guest OS per container — but every container on the host shares one kernel. A kernel exploit breaks the isolation. VMs are heavier, more isolated.

> **Exam trap:** Containers share the host OS; VMs each carry their own guest OS. "Three identical Linux guests on one hypervisor" is the inefficiency that motivates containers.

### IoT — Internet of Things

Sensors, smart-home devices, smart-watches, video doorbells, HVAC. Built by companies that are good at thermostats, not at security. Treat every IoT device as a potential pivot point and segment it (often onto its own VLAN with strict egress filtering).

### SCADA / ICS

Supervisory Control And Data Acquisition / Industrial Control System. Manufacturing, power generation, refineries, water treatment. Operators sit in a control room and command field equipment. The defining property: **must be heavily segmented from anything else**, because the consequences of compromise are physical.

### RTOS — real-time operating system

Deterministic — a critical task can preempt everything else and grab all resources instantly. Used in cars (ABS, airbags), aircraft, medical devices, military, factory PLCs. You don't run antivirus on these; you isolate them. Windows, Linux, macOS are **non-deterministic** by contrast.

### Embedded systems

Purpose-built hardware-plus-software, single-function, often no general-purpose OS access. Traffic lights, digital watches, medical monitors, HVAC controllers. The exam pairs these with the patching trap below.

### High availability (HA)

A configuration designed to keep running through a failure, not just to have a spare.

| | Redundant | High availability |
|---|---|---|
| Spare exists | Yes | Yes |
| Spare is hot | Maybe | Yes — already powered, configured, syncing state |
| Failover time | Manual / minutes | Automatic / seconds |

HA modes: **active-passive** (standby waits) vs **active-active** (both serve, capacity doubles, either can absorb the other's load on failure). HA costs more — duplicate firewalls, duplicate network paths, duplicate power. Eventually the cost crosses the value of the downtime it prevents, and that's where the business decision lives.

## Infrastructure considerations

The cross-cutting "-ilities" the exam will ask about by scenario:

| Consideration | What it means | Typical metric |
|---|---|---|
| **Availability** | System is up *for the right people* | Uptime % (e.g. 99.999%) |
| **Resilience** | How fast you recover from a failure | MTTR — Mean Time To Repair |
| **Cost** | Install + maintain + replace + tax/depreciation | TCO |
| **Responsiveness** | Latency between request and reply | App response time |
| **Scalability / elasticity** | Capacity grows and shrinks with load | Auto-scale events |
| **Ease of deployment** | How quickly an instance can be built | Time-to-deploy |
| **Risk transference** | Move financial risk to a third party | Cyber-insurance coverage |
| **Ease of recovery** | How quickly a broken system is rebuilt | Image-restore time vs OS-reinstall time |
| **Patch availability** | Can the system even be patched? | Vendor patch cadence |
| **Power** | Electrical supply, including backup (UPS, generator) | Watts, runtime |
| **Compute** | Processing capacity, scalable in cloud | vCPU, instance size |

### The patching trap

Embedded systems, IoT thermostats, HVAC controllers, time clocks, SCADA gear — vendors often release **no patches at all**. The control isn't "patch it"; it's **segment it behind a firewall, monitor it, and replace it eventually**. Exam scenarios that say "device cannot be patched" want this answer, not "patch it anyway."

### Scalability vs elasticity

**Scalability** = can grow. **Elasticity** = grows and shrinks automatically with load. Cloud auto-scaling is elasticity; spec'ing a server with extra cores is scalability. Both can be valid answers, but elasticity is the cloud-specific one.

### MTTR

Mean Time To Repair — how long to swap or restore a failed component. Often confused with MTBF (Mean Time Between Failures, a reliability metric for *how often* it breaks). The exam: MTTR = "how long broken." MTBF = "how often broken."

## How to read scenario questions

1. **Locate the workload.** On-prem? Cloud? Which service model? IoT/SCADA/embedded edge case?
2. **Locate the packaging.** Monolithic, VM, container, serverless, microservice?
3. **Name the trade-off.** What does this combination give up — patchability, isolation, cost, latency, control?
4. **If the question hands you a phrase like "while waiting for patches" or "until permanent fix" — pivot to the [[1.1 ✓|1.1]] control vocabulary (compensating, etc.).**
