# 4.5 — Enterprise Capabilities to Enhance Security

CompTIA description: *"Given a scenario, modify enterprise capabilities to enhance security."*

This is the big toolbelt objective: firewalls, web filtering, OS-level controls, secure transport protocols, email security (SPF/DKIM/DMARC), file/data monitoring (FIM, DLP), and endpoint security (AV, EDR, XDR). Seven sub-areas. The exam tests *which tool for which scenario* and the near-neighbor distinctions (forward vs reverse proxy, IDS vs IPS, AV vs EDR vs XDR, SPF vs DKIM vs DMARC, HTTP vs HTTPS).

---

## Firewalls

### Traditional firewall vs Next-Generation Firewall (NGFW)

| | Traditional | NGFW |
|---|---|---|
| Decision layer | L4 (port, IP) | L7 (application) |
| Inspection | Packet headers | Deep packet inspection |
| Other names | Packet filter | App-layer gateway, stateful multilayer, DPI |

Beyond filtering, firewalls commonly act as VPN concentrators and L3 routers (NAT, dynamic routing). The Ingress/Egress firewall typically separates Internet from Internal and often hosts a **screened subnet (DMZ)** for publicly reachable services.

### Rule processing

Rules evaluate **top-down**. First match wins. More-specific rules go up top, more-general rules below. Most firewalls have an **implicit deny** at the bottom — anything that doesn't match an allow rule is dropped.

### ACL parameters

Source IP, destination IP, source port, destination port, protocol, application (NGFW only), user, time-of-day, URL category — any subset can appear.

Standard port-to-service crib:

| Port / Protocol | Service |
|---|---|
| TCP 22 | SSH |
| TCP 80 | HTTP |
| TCP 443 | HTTPS |
| TCP 3389 | RDP |
| UDP 53 | DNS |
| UDP 123 | NTP |
| ICMP | ping / error messaging (no port) |

### IPS rule base — signatures and anomalies

IPSes ride inside or alongside firewalls today. Their rule base contains **signatures** (specific traffic patterns) and **anomaly rules** (generic patterns like SQL injection shapes). Thousands of signatures per IPS; rules are commonly grouped so you can set policy at the group level.

## Web filtering

### Goal

Control what data is seen in browsers and what content reaches users — block malware-hosting sites, restrict by category, enforce acceptable-use policies, prevent data exfiltration via web upload.

### Mechanisms

| Mechanism | Where it runs | Notes |
|---|---|---|
| **URL filter on NGFW** | In-line on the network | Works only when traffic crosses the firewall |
| **Agent-based content filter** | On each endpoint | Works anywhere — corporate net, home, coffee shop |
| **Proxy server** | In-line between client and Internet | Can cache, filter, log; explicit or transparent |
| **DNS filtering** | At the DNS resolver | Returns no IP (or a redirect IP) for bad domains |

### Forward vs reverse proxy

| | Forward proxy | Reverse proxy |
|---|---|---|
| Sits | In front of outbound clients | In front of inbound servers |
| Protects | Internal users from the internet | Internal servers from the internet |
| Visibility | Explicit (configured) or transparent (invisible) | Same axes apply |

> **Exam trap:** A forward proxy makes outbound requests on behalf of internal users. A reverse proxy receives inbound requests on behalf of internal servers. Direction = which side is being defended.

### URL categories and reputation

Most filters group URLs into categories (Adult, Gambling, Hacking, Education, Home & Garden, Travel, etc.). Filtering policies usually combine **category** with **reputation** — Trustworthy → allow; Low/Medium-risk → allow with logging; Suspicious/High-risk → block. Reputations are auto-assigned by scanners and manually overrideable.

### DNS filtering

The malicious-FQDN lookup fails — the resolver returns either no IP or a sinkhole IP. Works for any traffic that uses DNS (not just web), so it also catches malware reaching out to command-and-control servers.

## OS-level access controls

### Active Directory + Group Policy (Windows)

Active Directory is the central database of users, computers, file shares, printers, security groups. **Group Policy** overlays configuration and security rules on top of AD — login scripts, network settings (QoS), security parameters. Managed centrally through the Group Policy Management Editor; pushed to every domain-joined device.

### SELinux — mandatory access control on Linux

Default Linux uses **discretionary access control (DAC)** — users grant access at their own discretion to objects they own. **SELinux** (Security-Enhanced Linux) adds **mandatory access control (MAC)** — a central administrator defines policy, users can't override it. Open-source, installable on most distributions, used in highly secure environments. Implements **least privilege** by default.

## Secure protocols — replace cleartext

Default to encrypted. Telnet, FTP, SMTP, IMAP, POP3, HTTP all transmit in cleartext. Pick their secure replacements:

| Insecure | Secure | Port (insecure → secure) |
|---|---|---|
| Telnet | SSH | 23 → 22 |
| HTTP | HTTPS | 80 → 443 |
| FTP | SFTP (or FTPS) | 21 → 22 (SFTP) |
| IMAP | IMAPS | 143 → 993 |
| POP3 | POP3S | 110 → 995 |
| SMTP | SMTPS | 25 → 465 / 587 |

> **Exam trap:** A given port number is a hint, not a guarantee. Port 443 *usually* means HTTPS, but a misconfigured server can run anything there. Capture traffic to confirm encryption is actually in use.

### Tunnel-level encryption — VPN

When the application itself can't be made secure, encrypt the network layer. A **VPN** (objective 3.2) creates an encrypted tunnel between client and concentrator; everything inside is protected regardless of the app's native security.

## Email security — SPF, DKIM, DMARC

Email's underlying protocols have minimal built-in authentication. To stop spoofing, three DNS-based mechanisms layer on:

### The mail gateway

The gatekeeper. Receives mail, checks SPF/DKIM/DMARC, decides delivery (inbox vs spam vs reject). On-prem in a screened subnet, or cloud-hosted as a service.

### SPF — Sender Policy Framework

A **DNS TXT record** for your domain lists the servers authorized to send mail on its behalf. Receiving mail servers query this record to verify the sending IP is approved.

```
v=spf1 include:mailgun.org -all
```

### DKIM — DomainKeys Identified Mail

Outgoing mail servers attach a **digital signature** in the email headers using a private key. The matching **public key is published as a DNS TXT record**. Receivers fetch the public key, verify the signature, confirm the message wasn't tampered with and really came from a server with that domain's private key.

DKIM is a transport-level signature between mail servers, **not** a S/MIME-style signature visible in the mail body.

### DMARC — Domain-based Message Authentication, Reporting, and Conformance

Builds on SPF and DKIM. Another DNS TXT record tells the receiving server what to do when SPF or DKIM fails: **accept**, **quarantine** (spam folder), or **reject**.

```
v=DMARC1; p=quarantine; rua=mailto:dmarc-reports@example.com
```

DMARC also specifies a **reporting destination** — receivers send back compliance reports showing how many messages purported to be from your domain, how many passed, how many were quarantined or rejected. Lets you detect spoofing campaigns against your domain.

> **Exam trap:** Three roles, easy to confuse.
> - **SPF** = which servers are allowed to send (sender authorization)
> - **DKIM** = signature on each message (per-message integrity + authenticity)
> - **DMARC** = what to do when SPF/DKIM fail + where to send reports

## Monitoring data — FIM and DLP

### File Integrity Monitoring (FIM)

Watches application binaries, libraries, and critical OS files for unexpected change. Tools:

| OS | Tool |
|---|---|
| Windows | **System File Checker (sfc)** — on-demand check, replaces bad with good |
| Linux | **Tripwire** — continuous monitoring with real-time alerts |
| Cross-platform | Host-based IPS (HIPS) often includes FIM |

### DLP states

DLP works at three states of data — match the state to the deployment:

| State | Where data is | Where DLP runs |
|---|---|---|
| **Data in use** | Active memory of a workstation | Endpoint DLP software |
| **Data in motion** | Crossing the network | Network DLP appliance / NGFW-integrated |
| **Data at rest** | Stored on disk | DLP software on the file/database server |

### Email DLP and USB DLP

**Email DLP** scans outbound (and inbound) email for sensitive content — Social Security numbers, W-2 data, fake wire-transfer phishing replies. The 2016 Boeing case: an employee emailed a spreadsheet to their spouse; hidden cells contained 36,000 employees' PII. Email DLP would have caught it.

**USB DLP** restricts use of removable storage. The 2008 US DoD agent.btz worm spread via USB drives, leading to a ban on all USB removable media on DoD networks until 2010.

### Cloud DLP

DLP for SaaS and IaaS — Salesforce, M365, AWS S3, etc. Watches sensitive data going into cloud apps, blocks before it persists.

## Endpoint security — AV vs EDR vs XDR

### Antivirus / antimalware

Signature-based detection of known threats. Necessary, not sufficient. Modern malware variants are produced by the millions per day — signature catalogs can't keep up alone.

### EDR — Endpoint Detection and Response

Adds **behavioral analysis** and **machine learning** to signatures. Runs as an agent on the endpoint. When it sees suspicious activity:

- Auto-isolate the malicious process
- Quarantine the malware
- Roll back the system to a known-good state
- Provide **root-cause analysis** — *how* did this get in?

### XDR — Extended Detection and Response

Same idea as EDR but **correlates across many endpoints plus network telemetry plus identity systems**. Single-endpoint blind spots disappear. Adds:

- **User Behavior Analytics (UBA)** — baseline of normal user activity, alert on deviation
- Cross-system correlation (a login here + a file access there + an outbound flow over there = one attack)
- Faster investigation through automation

> **Exam trap:** **AV** finds known signatures. **EDR** adds behavioral analysis and per-endpoint response with root-cause analysis. **XDR** scales EDR to correlate across many endpoints, networks, and identities. Each is a superset of the previous.

### Posture assessment and NAC

Before letting a device on the network, check it. Look for: trusted device cert installed, AV running and up to date, OS patched, applications current, full-disk encryption enabled. Failures get **quarantined** (own VLAN, remediation instructions), not granted full access.

Three agent models for posture checks (same axis as monitoring):

- **Persistent agent** — always running on the device
- **Dissolvable agent** — runs at login, removes itself after
- **Agentless** — Active-Directory-integrated, runs on login/logoff only

## How to read scenario questions

1. **What's the goal?** Block app traffic (NGFW), block web content (URL filter / DNS filter), block sensitive data (DLP), block bad email (mail gateway + SPF/DKIM/DMARC), inspect endpoint (AV/EDR/XDR).
2. **What's the data state?** In use → endpoint DLP. In motion → network DLP. At rest → server DLP.
3. **Email auth failure?** SPF = who can send. DKIM = signature on message. DMARC = what to do when SPF/DKIM fail + reporting.
4. **Endpoint protection scope?** Single endpoint signatures = AV. Single endpoint with behavior + rollback = EDR. Multi-source correlation = XDR.
5. **Forward or reverse proxy?** Outbound user protection = forward. Inbound server protection = reverse.
