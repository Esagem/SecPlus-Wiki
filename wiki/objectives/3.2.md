# 3.2 — Secure Enterprise Infrastructure

CompTIA description: *"Given a scenario, apply security principles to secure enterprise infrastructure."*

This is the "boxes on the network diagram" objective. Six categories of device or technique — zones, IDS/IPS, appliances, port security, firewalls, secure communications — and the exam asks you to pick the right one for the scenario, or to distinguish near-neighbors (IDS vs IPS, forward vs reverse proxy, UTM vs NGFW, IPsec vs SSL VPN, SD-WAN vs SASE).

---

## Secure infrastructures — zones and attack surface

### Security zones

A **security zone** is a logical label applied to part of the network based on use and trust, not on IP range. Rules are written between zones, not between addresses — easier to read and easier to audit at scale.

| Zone name | Meaning |
|---|---|
| Trusted / Internal | Inside the corporate boundary, default-deny from outside |
| Untrusted / Internet | Outside, hostile by default |
| Screened / DMZ | Publicly reachable services (web, mail, DNS) — Internet can reach in, but the zone can't freely reach Internal |
| Database / Servers | Tighter than Internal, only accessible from app tier |

Typical rules read as **source-zone → destination-zone**: "Trusted → Untrusted: allow web." "Untrusted → Screened: allow HTTPS to web servers."

### Attack surface

The full set of openings an attacker could use — open ports, vulnerable application code, weak auth flows, human error in firewall rules. The job is to shrink it: code audits, port closure, real-time monitoring, encryption of cabling/traffic so even an internal tap can't read packets.

## IDS vs IPS

| | IDS | IPS |
|---|---|---|
| Action | Detects and alerts | Detects, alerts, **blocks** |
| Position | Out-of-band (passive) | In-line (active) |
| Effect on traffic | None — copy is inspected | Sits in the path; can drop |
| Failure | No impact on traffic | Behavior depends on fail-open / fail-closed |

> **Exam trap:** "Watches and logs but doesn't block" = IDS. "Watches and blocks in real time" = IPS. The P stands for prevent.

### Active (inline) vs passive monitoring

**Inline / active monitoring** — traffic flows *through* the device. Can block, but a failure can take down the network depending on fail mode.

**Passive monitoring** — the switch sends a *copy* of traffic to the sensor via a **port mirror** (also called **SPAN** — Switch Port Analyzer) or a physical **network tap**. Cannot block in real time, only alert. Even an IPS deployed passively functions only as an IDS.

### Fail-open vs fail-closed

When an inline device crashes:

| Mode | What happens | Trade-off |
|---|---|---|
| **Fail-open** | Traffic continues to flow, no inspection | Availability preserved, security gap during outage |
| **Fail-closed** | Link is severed | Security preserved, network goes down |

Most production networks pick fail-open for availability. High-security environments may pick fail-closed. Check the vendor docs — defaults vary.

## Network appliances

### Jump server

A hardened internal host that external admins must connect to first; from there they SSH/RDP to internal targets. A single audited choke point for remote admin. If the jump server is compromised, the inside is exposed — harden it accordingly (MFA, minimal services, full logging).

### Proxy server

Sits between client and target, making the request on the client's behalf. Adds caching, URL filtering, content scanning.

| Axis | Variants |
|---|---|
| **Direction** | Forward (internal clients → Internet) · Reverse (Internet clients → internal server) |
| **Visibility to client** | Explicit (configured in OS/browser) · Transparent (client doesn't know) |
| **Trust** | Corporate-managed · Open (anyone-can-use, third party — security concern, often blocked) |

> **Exam trap:** Forward proxy protects outbound users from the internet. Reverse proxy protects inbound servers from the internet. The direction tells you which side is being defended.

### Load balancer

Distributes incoming traffic across a pool of servers. Detects failed servers and removes them from rotation.

| Mode | Behavior |
|---|---|
| **Active-active** | All pool members handle traffic; capacity scales with pool size |
| **Active-passive** | Some members serve; standbys are idle until an active fails |

Extras: **TCP offload** (one persistent TCP session per backend instead of one per client), **SSL/TLS offload** (decrypt at the LB, send plaintext to backends), **caching**, **content switching** (route by URL/path), **prioritization** (QoS).

### Sensors and collectors

**Sensors** gather data — sometimes built into switches/routers/firewalls/IPSes, sometimes purpose-built network taps. **Collectors** consolidate it. A **SIEM** (Security Information and Event Manager) is the canonical collector — single database, correlation rules, alerting, reporting across diverse sources.

## Port security — 802.1X and EAP

The control that gates *physical* network access. Before a device can talk on the wired or wireless network, it authenticates.

| Component | Role |
|---|---|
| **EAP** (Extensible Authentication Protocol) | The framework — defines the conversation, not the credentials |
| **802.1X** | The IEEE standard that runs EAP over a switch port or wireless association — aka **port-based NAC** |
| **Supplicant** | The client trying to connect |
| **Authenticator** | The switch port / access point / wireless controller |
| **Authentication server** | The credential store (RADIUS, TACACS+, LDAP/Active Directory via Kerberos) |

Flow: supplicant connects → authenticator opens an EAP request → supplicant responds → authenticator forwards to auth server → auth server says yes/no → port opens or stays closed.

> **Exam trap:** 802.1X is not the database — it's the protocol that talks *to* the database. The database is RADIUS/TACACS+/AD/etc. Mixing those layers is a common wrong answer.

## Firewall types

### The four classes

| Type | OSI layer | Decision basis | Typical role |
|---|---|---|---|
| **Network firewall (traditional)** | L4 | Port and IP | Allow/deny by 5-tuple |
| **UTM** (Unified Threat Management) | L4 mostly | Port + bundled features (URL filter, AV, IDS, spam, VPN, QoS, CSU/DSU) | Older "all-in-one" appliance |
| **Next-Generation Firewall (NGFW)** | L7 | **Application** identity via deep packet inspection | Modern primary perimeter |
| **WAF** (Web Application Firewall) | L7 (HTTP/HTTPS only) | Inspects HTTP payloads | Sits in front of web apps, blocks SQLi/XSS |

### UTM vs NGFW

UTM bundles many features into one box but typically inspects at L4 — port-based, with bolt-on features. Turning on every UTM feature usually crushes throughput. NGFW is *built* around L7 application identification: it knows the traffic is "Twitter post" vs "Twitter view" vs "Microsoft SQL Server" regardless of port. NGFW commonly absorbs the IPS function and URL categorization too.

> **Exam trap:** "Allow Twitter view but block Twitter post" → NGFW. "Bundle of features in a single box" → UTM. "Blocks SQL injection against the web app" → WAF.

### WAF

Specialized for web traffic. Inspects HTTP/HTTPS payloads and blocks application-layer attacks (SQLi, XSS, command injection). Often **required** by compliance — PCI-DSS mandates a WAF or equivalent for systems handling cardholder data. Pairs naturally with an NGFW — they inspect different things.

## Secure communication

### VPN — the basics

A VPN encrypts private traffic so it can cross a public network safely. A **VPN concentrator** is the endpoint device — often a function of an NGFW today, sometimes a dedicated appliance, sometimes pure software.

### IPsec — site-to-site

Used to bind two networks together permanently (branch ↔ HQ, on-prem ↔ cloud). Endpoints are usually firewalls; user devices need no special config.

The IPsec packet structure, simplified:

```
[ new IP header | IPsec header | ENCRYPTED( original IP header | data ) | IPsec trailer ]
```

The original packet is encrypted and **tunneled** inside a new IP header that points to the far-side concentrator. Routers along the way see only the outer header.

### SSL/TLS VPN — remote access

Used for individual remote users connecting from laptops, tablets, phones. Runs over TCP **443**, so it traverses firewalls trivially. Client is software on the device (sometimes built into the browser or OS). Credentials are normal user auth; can be **always-on**.

| | IPsec VPN | SSL/TLS VPN |
|---|---|---|
| Typical use | Site-to-site | Remote access |
| Endpoints | Firewall ↔ Firewall | User device ↔ Concentrator |
| Port / Protocol | UDP 500/4500, ESP | TCP 443 |
| Firewall friendliness | Sometimes needs ESP/IKE exemptions | Goes through anything that allows HTTPS |

> **Exam trap:** "Remote employee on a laptop" → SSL/TLS VPN. "Branch office talking to HQ" → IPsec site-to-site. The exam loves this pairing.

### SD-WAN

Software-defined wide-area networking. Built for the world where applications live in multiple clouds, not in one central data center. SD-WAN dynamically chooses the best path per application from a remote site — to HQ, or directly to the cloud SaaS, or to a particular IaaS region — without forcing every packet through a central data center.

### SASE — Secure Access Service Edge

The "next generation of VPN." Pushes security functions (NGFW, secure web gateway, ZTNA, CASB) into the cloud, next to the apps. A SASE client on the device connects securely into the SASE cloud, which then brokers access to whatever app the user needs — corporate, SaaS, or otherwise.

> **Exam trap:** SD-WAN is about *connectivity* across distributed sites and clouds. SASE is about *security* layered on top of that connectivity. Many deployments use both: SD-WAN for the network, SASE for the security overlay.

## How to read scenario questions

1. **What's the goal?** Segment a network? Authenticate a port? Inspect an application protocol? Connect a remote user?
2. **Match to the right layer.** Port number → L4 firewall. Application name → NGFW. HTTP payload → WAF. Switch port auth → 802.1X.
3. **Active or passive?** If the device must *block*, it must be in-line — IPS, NGFW, WAF. If it only *alerts*, IDS or port mirror suffices.
4. **Remote vs site?** Single laptop = SSL/TLS VPN. Branch to HQ = IPsec site-to-site.
