# 3.4 — Resilience and Recovery

CompTIA description: *"Explain the importance of resilience and recovery in security architecture."*

This objective covers how to keep services running when something breaks — at the component, site, application, and grid-power level — and how to recover when prevention fails. The exam tests recognition: given a failure scenario, which technique fits.

---

## The framework

Resilience layers, smallest to largest:

| Layer | Mechanism |
|---|---|
| **Component** | Redundant device, HA pair, clustering, load balancing |
| **Site** | Hot / warm / cold recovery site, geographic dispersion |
| **Platform** | Platform diversity (OS mix), multi-cloud |
| **Data** | Backups, snapshots, replication, journaling |
| **Power** | UPS, generator |
| **Process** | Capacity planning, recovery testing, continuity of operations (COOP) |

---

## Resiliency

### High availability vs simple redundancy

| | Redundant | High availability (HA) |
|---|---|---|
| Spare exists? | Yes | Yes |
| Spare is on? | Maybe (in a closet) | Yes — powered, configured, syncing state |
| Failover | Manual / hours | Automatic / seconds |

> **Exam trap:** "We bought a backup unit and put it on the shelf" → redundant, not HA. HA means the spare is hot and ready.

### Server clustering vs load balancing

Both spread work across multiple servers, but the architecture differs:

| | Server cluster | Load balancer |
|---|---|---|
| Awareness | Each node knows about the others | Servers don't know about each other |
| Coordinator | Distributed (cluster software) | Central (the LB) |
| Storage | Often shared storage so all nodes see the same data | Each server typically has its own state |
| OS uniformity | Usually identical OS across cluster | Can be mixed — LB doesn't care |

### Site resiliency — hot, warm, cold

| Site | Hardware on-site | Data on-site | Recovery time | Cost |
|---|---|---|---|---|
| **Hot** | Full duplicate | Continuously synced | Hours or less | Highest |
| **Warm** | Some hardware | Some data, possibly stale | Days | Medium |
| **Cold** | Empty building with power | None | Weeks | Lowest |

> **Exam trap:** "Empty building, just power and lighting" → cold. "Exact replica, ready to take over today" → hot. "Some equipment, some data, more setup needed" → warm.

### Geographic dispersion

Recovery site far enough from the primary that a single disaster (hurricane, flood, regional power outage) won't hit both. The further apart, the harder logistics get — transporting equipment, getting staff on site, eventual cutover back — so distance is balanced against operational feasibility.

### Platform diversity

Run a mix of operating systems and platforms so a single vendor vulnerability doesn't take everything down. Windows + Linux in the data center, mixed clients, etc. Trades single-vendor simplicity for blast-radius reduction.

### Multi-cloud

The platform-diversity logic at the cloud level. AWS + Azure + GCP — when one provider has an outage, the others are usually unaffected. Adds operational complexity (different APIs, IAM models, log formats) for resilience benefit.

### Continuity of operations (COOP)

The manual / non-technical fallback when nothing technological is working. Paper receipts, phone-in card approvals, hand-signed transactions. The exam wants you to recognize that COOP is **planned in advance** — when the disaster hits, it's too late to design.

## Capacity planning

The discipline of matching supply (people, technology, infrastructure) to demand. Three dimensions:

| Dimension | "Too little" cost | "Too much" cost |
|---|---|---|
| **People** | Overworked staff, slow service | Payroll waste, layoffs |
| **Technology** | Slow systems, outages under load | Spend on unused capacity |
| **Infrastructure** | Can't scale when needed | Idle racks, idle cloud spend |

People are the hardest to scale — hiring and training takes weeks. Cloud is the easiest — click to add capacity. Choose technology that *can* scale: load balancers with multiple servers, partitioned databases, cloud-native services that grow and shrink on demand.

## Recovery testing

You only know your recovery plan works if you've tested it. Four kinds of test, in increasing intensity:

| Test | What happens | Cost / risk |
|---|---|---|
| **Tabletop exercise** | Walk through the steps verbally around a table | Cheap, low risk, finds plan gaps |
| **Simulation** | Run a scripted event (phishing email, password-reset test, data-exfil test) without touching production | Medium |
| **Failover test** | Trigger an actual failover to redundant systems | Higher — risk of disrupting users |
| **Full recovery test** | Cut over to recovery site as if disaster were real | Highest cost and risk |

### Tabletop exercise

Cheapest, most-used recovery test. The team walks through "imagine X just happened" and traces each step they'd take. Catches missing assumptions and process gaps before a real incident. No actual systems are touched.

### Failover test

Designed to verify that redundant configurations actually fail over. Pull the cable on the primary firewall and watch the secondary take traffic. Routers, firewalls, switches with built-in HA all have failover features; this confirms they fire.

### Phishing simulation

A specific simulation kind. Craft a phishing email, send to employees, measure how many click. Tests both technical filters and human awareness. Click rates feed into [[5.6 ✓|user training]] decisions.

### Parallel processing

Resilience pattern, not strictly a test — multiple CPUs or hosts process transactions simultaneously. If one fails, the others absorb the load.

## Backups

### Variables to plan

Every backup design picks values for these:

| Variable | Choices |
|---|---|
| **How much** | Just changes? Whole system? Specific datasets? |
| **Backup type** | Full · Incremental · Differential · Snapshot |
| **Media** | Tape · Disk · Cloud |
| **Location** | On-site · Off-site · Cloud |
| **Schedule** | Hourly · Daily · Weekly · Monthly |
| **Encryption** | Mandatory for off-site and cloud |
| **Restore testing** | Untested backups are a guess |

### On-site vs off-site

| | On-site | Off-site |
|---|---|---|
| Speed of restore | Fast | Slower (transit) |
| Disaster survival | Lost if facility is lost | Survives local disaster |
| Cost | Lower | Higher (storage fees, transport) |
| Common pattern | Combine: on-site for fast recovery, off-site for disaster |

> **Exam trap:** A backup tape stolen out of an employee's car while they ran errands → exactly why off-site backups must be encrypted.

### Snapshots

Point-in-time capture of a system, typically a VM or cloud instance. Easy and fast — one click. After the first full snapshot, subsequent snapshots capture only the *changes*, much like an incremental backup. Used routinely before risky changes and as a daily backup mechanism for VMs.

### Replication

Near-real-time copy of data to one or more remote locations. Source change → replica updated in seconds. Used for hot sites (instant cutover), backup storage, and read-scale architectures. Replicated data is "live enough" to take over immediately.

### Journaling

Protects against corruption from power loss or crash *during a write*. The data is first written to a journal on disk; once journal write completes, it's applied to the database. If power drops mid-process, on restart the system replays the journal to recover. File systems (NTFS, ext4) and databases both use journaling.

### Restore testing — non-negotiable

Untested backups are not backups; they are wishes. Test periodically: simulate a disaster, restore the data, and verify applications can actually use it. Restoration is only half the test — the other half is whether the application reads it correctly.

## Power resiliency

### Power problems

| Event | What's happening |
|---|---|
| **Blackout** | Complete loss of power |
| **Brownout** | Sustained drop in voltage |
| **Surge** | Excessive voltage spike |

### UPS — Uninterruptible Power Supply

Battery-backed power for short outages (minutes to a couple of hours). Three classes:

| Type | Behavior | Use case |
|---|---|---|
| **Offline / Standby** | Runs on main power, switches to battery on failure | Workstations, light servers |
| **Line-interactive** | Adjusts voltage during brownouts without switching to battery | Areas with frequent brownouts |
| **Online / Double-conversion** | Always running from battery (main power continuously recharges) | Critical equipment — no switchover delay at all |

UPSes can also signal connected systems to **shut down gracefully** before the battery dies.

### Generator

Long-term power backup. Burns fuel — runs as long as fuel lasts. Sized for the building or just for designated outlets (often visibly marked).

### UPS + generator together

The standard data center pattern. Generator takes ~30–60 seconds to start and stabilize. The UPS bridges that gap — keeps everything running through the start-up window, then the generator carries the load until grid power returns.

> **Exam trap:** "Need to cover the gap while the generator starts" → UPS. "Need to run for hours or days during an extended outage" → generator. They aren't substitutes; they pair.

## How to read scenario questions

1. **What failed?** Component, site, platform, data, or power?
2. **How fast must we recover?** Seconds (HA, replication, hot site) → minutes (failover) → hours (warm site, snapshot restore) → days (cold site, tape restore).
3. **Is this prevention or recovery?** HA/clustering/redundant power = prevention of downtime. Backups/recovery site = recovery from downtime.
4. **If the answer involves both speed AND distance from the primary**, expect "hot site + replication." If speed matters less than cost, expect "warm" or "cold."
