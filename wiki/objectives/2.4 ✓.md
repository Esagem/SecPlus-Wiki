# 2.4 — Malicious Activity

CompTIA description: *"Given a scenario, analyze indicators of malicious activity."*

This is the largest objective in Domain 2 — fifteen videos covering every named attack and malware type on the exam. The exam pattern is consistent: a scenario describes behavior, and you classify it. Memorize the *signature behavior* of each named item below and you can answer the questions cold.

---

## The malicious-activity map

| Family | What it does |
|---|---|
| **Malware** | Software designed to harm — viruses, worms, ransomware, trojans, rootkits, spyware, keyloggers, logic bombs, bloatware |
| **Physical attacks** | Brute force, RFID cloning, environmental attacks |
| **Network attacks** | DoS / DDoS, DNS attacks, wireless attacks, on-path attacks |
| **Replay attacks** | Pass-the-hash, session hijacking |
| **Malicious code** | The class containing injection, scripts, exploits delivered through code |
| **Application attacks** | Injection, buffer overflow, privilege escalation, CSRF, directory traversal |
| **Cryptographic attacks** | Collisions, downgrade attacks |
| **Password attacks** | Plaintext storage, spraying, brute force |
| **IOCs** | The evidence that ANY of the above happened |

---

## Malware

### Virus
Self-replicating code that requires **user action** to start (clicking a link, running an executable). Spreads through file systems and across the network once it's running. Subtypes the exam names:

| Subtype | Mechanism |
|---|---|
| **Boot sector virus** | Runs when the system boots, before the OS loads |
| **Macro virus** | Lives inside a Microsoft Office document and runs via the macro engine |
| **Script-based virus** | Runs in a scripting host (PowerShell, JavaScript, etc.) |
| **Fileless virus** | Never writes to disk — lives only in memory, often persisted via a registry auto-start that re-injects on boot |

### Worm
Self-replicating like a virus but **needs no user action** to spread. Worms propagate at network speed. **WannaCry** is the canonical example — used the EternalBlue SMB vulnerability to spread automatically, then installed ransomware on each infected host.

### Ransomware
Encrypts the victim's data and demands payment for the decryption key. Often delivered by a worm or via phishing → trojan → install. The defenses to memorize: **offline backups**, patched OS, patched apps, AV/anti-malware.

### Trojan horse
Software that appears to do one thing (game, utility) but secretly does something malicious. Requires user installation.

### RAT (Remote Access Trojan)
A trojan that gives the attacker remote control. Often includes a keylogger; the transcript names **Dark Comet** as a sample tool.

### Spyware
Watches the user — browsing habits, keystrokes, screen content, clipboard, IM chats — and exfiltrates the data. Often arrives bundled with peer-to-peer software, fake security tools, or malicious email links.

### Bloatware
Pre-installed manufacturer apps you didn't ask for. Not always malicious, but: takes storage, may auto-start (performance), and any unknown app is a possible vulnerability. Remove via the OS's uninstall mechanism or, if stubborn, third-party uninstall tools.

### Keylogger
Captures every keystroke and exfiltrates it. Often part of spyware, RATs, or trojans. Sometimes goes further — clipboard captures, screenshots, search-query logs. Encryption protects data in transit and at rest; the keyboard input itself isn't encrypted, which is why keyloggers are valuable to attackers.

### Logic bomb
Malicious code that detonates on a trigger — a date/time, a particular user logging in, a file change. The 2013 South Korean bank/broadcasting logic bomb detonated at 2 PM local time on March 20, wiping master boot records and bricking ATMs.

Logic bombs are **hard to detect** because they're often custom-written for one target — no AV signature. Defenses: file-integrity monitoring on critical OS files, change-control processes, least privilege so installation requires admin access.

### Rootkit
Hides itself in the OS kernel (or sometimes as a normal process). Once installed, it's effectively part of the OS — invisible to traditional process listings and AV. The exam-named defense: **Secure Boot** (UEFI). It validates the OS kernel signature at boot and refuses to load a modified kernel.

## Physical attacks

| Attack | Mechanism |
|---|---|
| **Brute force (physical)** | Forcing a door or window. Not a password attack — physical access yields full control. |
| **RFID cloning** | Reading a victim's RFID badge and writing a duplicate. Cloner devices cost <$50. Defense: MFA — a cloned badge can't supply the user's PIN or biometric. |
| **Environmental attacks** | Targeting infrastructure around the data center: power, HVAC, fire suppression. Often easier than attacking the systems directly. |

---

## Denial of service (DoS)

A **DoS** forces a service to fail — by overloading it, exploiting a vulnerability, or just unplugging it. A **DDoS** uses many distributed sources (a **botnet**) to overwhelm the target. Zeus had ~3.6M bots at its peak.

### Amplification / reflection
The attacker uses an internet service that returns a *bigger* response than the request, with the victim's address spoofed as the source. DNS amplification (~86× factor), NTP amplification, and ICMP-based amplification are the named examples.

```
Attacker → spoofed-source query → Open DNS resolver
Open DNS resolver → large response → Victim
```

Small request, large response, victim swamped. The attacker stays cheap.

---

## DNS attacks

### DNS poisoning / cache poisoning
Modifying DNS records or cache so a name resolves to an attacker-controlled IP. Several flavors:

- **Modify the server itself** (requires compromising the DNS server — well-protected).
- **Modify the client's hosts file** (requires elevated access on the client).
- **MITM the resolution** (intercept DNS queries on the network and respond first).

The 2016 Brazilian bank breach saw attackers modify domain-registration settings for ~36 domains for six hours on a weekend, redirecting bank traffic to attacker servers.

### URL hijacking / typo squatting / brand jacking
Already covered under 2.2 as a delivery vector — same names appear under DNS attacks here because they manipulate the name → destination mapping. `professormessor.com` (note the spelling), `professormesser.org`, missing or doubled letters.

## Wireless attacks

### Deauthentication attack
Sends spoofed deauthentication management frames to force a client off a Wi-Fi network. A denial-of-service. Worked because pre-802.11ac wireless specs did NOT encrypt management frames. 802.11ac and newer encrypt **disassociate, deauthenticate, and channel-switch** frames by default, blocking this attack.

### RF jamming
Transmits noise on the same frequencies as the target wireless network, dropping the signal-to-noise ratio enough that clients can't communicate. Can be unintentional (microwave ovens, fluorescent lights interfering with 2.4 GHz) or intentional. Located via **fox hunting** with a directional antenna.

---

## On-path attacks

Older name: **man-in-the-middle**. The attacker sits between two parties, can read and modify traffic in transit, and remains invisible to both endpoints.

### ARP poisoning
A specific on-path technique on a local subnet. ARP has no authentication, so the attacker sends unsolicited ARP replies advertising their MAC as the gateway's IP. Victim caches the spoofed mapping and starts sending traffic to the attacker. Same technique works in reverse to MITM the gateway.

### On-path browser attack (man-in-the-browser, MITB)
Same idea but on the victim's *own machine* — malware acts as a proxy inside the browser. Since it operates inside the browser, **encryption doesn't protect against it** — the attacker sees data after decryption (outbound) or before encryption (inbound).

---

## Replay attacks

The attacker captures legitimate traffic and re-sends it to gain unauthorized access. Replay is often paired with an on-path attack to collect the material.

### Pass-the-hash
The attacker captures the hashed password sent during authentication and replays the same hash to the server. The server validates the hash and authenticates the attacker. Defenses: **encrypt the authentication channel**, **salt the password** (different salt → different hash each time), **reject reused hashes**.

### Session hijacking / sidejacking
The attacker steals the victim's session ID (cookie) and replays it. Server treats every request as coming from the legitimate authenticated user. Captured via wire-shark/packet capture, XSS, or browser extensions. Defenses: **HTTPS end-to-end**, secure-cookie flags, short session lifetimes.

---

## Malicious code

A catch-all for code-based attacks: executable malware, scripts, macros, trojan horses. Defenses are layered:

- Anti-malware to block known signatures
- Firewall to block known-malicious traffic
- Patching to close exploitable vulnerabilities
- User training to prevent the social half

Named exam examples: WannaCry (SMB v1 RCE → ransomware), British Airways (XSS injected into checkout pages → 380,000 cards), Estonian Central Health (SQLi → entire national database).

## Application attacks

Most of these overlap with 2.3 vulnerabilities — same mechanisms framed as the *attack* rather than the *flaw*.

| Attack | What it does |
|---|---|
| **Injection** | Adds malicious code to input. SQL, HTML, XML, LDAP. |
| **Buffer overflow** | Writes past the end of a buffer. |
| **Privilege escalation** | Uses a flaw to gain higher rights than the user is meant to have. Vertical = user → admin. Horizontal = user A → user B. |
| **Cross-site request forgery (CSRF / XSRF)** | Uses the victim's authenticated session to perform unintended actions on a trusted site. Defense: per-request anti-forgery tokens. |
| **Directory traversal** | Manipulates file paths (`../../`) to access files outside the web server's intended directory. |
| **Replay** | (Covered above.) |

**CSRF vs XSS:** Both use the browser. XSS runs *attacker script in trusted site context*. CSRF makes the *victim's browser send forged requests* using its existing authentication. Different mechanisms; both abuse browser trust.

## Cryptographic attacks

### Birthday attack / hash collision
Based on the birthday paradox — in 23 people, ~50% chance two share a birthday. Same math applies to finding two inputs that produce the same hash. MD5 has known collisions (researchers forged a CA-signed certificate in 2008); SHA-1 has been broken; both are deprecated. Defense: **use longer hashes (SHA-2/SHA-3)** and **deprecate broken algorithms**.

### Downgrade attack (e.g., SSL stripping)
The attacker — typically on-path — manipulates the negotiation between client and server so they use a weaker algorithm or no encryption at all. **SSL stripping**: the user requests `http://`, the server tries to redirect to `https://`, the attacker swallows the redirect and proxies `http://` to the user while speaking `https://` to the server. The user's traffic is unencrypted on their leg. Defenses: HSTS, HTTPS-only, certificate pinning.

## Password attacks

### Plaintext storage
The worst case — passwords stored as the literal text. A breach of the password file gives the attacker every credential immediately. Mitigation: stop using the application; properly: store **hashes**, ideally with salt and a slow hashing algorithm (bcrypt, Argon2).

### Password spraying
Attacker tries a *small number* of very common passwords (`123456`, `password`, `qwerty`) against a *large number* of accounts. Stays under lockout thresholds, surfaces the weakest accounts in the org. Most lockout policies don't trigger because each account only sees 1–3 attempts.

### Brute force
Tries every combination. **Online** brute force is slow (lockouts, rate limits). **Offline** brute force is fast — the attacker has the password file/database and runs a local job through GPU/ASIC clusters until the hash matches. This is why password hashes should be **slow** (computationally expensive) and **salted** (prevents precomputed rainbow tables).

**Spraying vs brute force:** spraying is *few passwords, many accounts*; brute force is *one account, many passwords*. The defenses differ: spraying calls for anomaly detection across accounts; brute force calls for slow salted hashes and account lockouts.

## Indicators of Compromise (IOCs)

The evidence that something happened. Memorize the named indicators — the exam loves to ask "which of these IOCs suggests X?"

| Indicator | What it suggests |
|---|---|
| **Account lockout (without user's attempted logins)** | Someone is trying to brute force the account, or pretexting a help-desk reset |
| **Concurrent sessions / impossible travel** | Same account logged in from two distant geographies in a short window |
| **Blocked content** (security sites, AV updates unreachable) | Malware blocking patches/updates to maintain persistence |
| **Resource consumption spikes** | Outbound data transfer, unexpected CPU/memory use — often exfiltration |
| **Resource inaccessibility** | Server crashed, files encrypted (ransomware), account locked |
| **Out-of-cycle logging** | Activity logged at times work shouldn't be happening |
| **Missing logs** | Attackers delete logs to hide their tracks. Sometimes the *absence* is the IOC. |
| **Published / leaked data** | Your organization's data appears on the internet — often the only signal that exfiltration occurred |

**Exam trap:** IOCs are *evidence*, not the attack itself. A question describing "the SIEM noticed 4 GB outbound to an unknown IP at 3 AM" is asking which IOC category — that's resource consumption, not the name of the attack that caused it.

---

## How to read scenario questions

1. **What's the observable behavior?** Encrypted files = ransomware. Self-spreading without user action = worm. Locked account with no user attempts = brute force or password spray indicator.
2. **For named-attack questions, look for the mechanism keyword.** Amplification → reflected DoS. Deauth frames → wireless deauth. ARP cache poisoned → ARP poisoning on-path. Hash captured and replayed → pass-the-hash.
3. **Distinguish similar pairs:**
   - Virus vs worm → user action vs no user action.
   - Spraying vs brute force → few×many vs many×one.
   - CSRF vs XSS → forged request via existing session vs malicious script in trusted page.
   - Replay vs on-path → replay reuses captured material; on-path is live MITM.
4. **For IOC questions:** the answer is the named *category* of evidence, not the attack itself.
