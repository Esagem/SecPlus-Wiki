# After-Action Review ‚Äî Wiki Build & 2026-05-25 Corruption Incident

## TL;DR

The wiki is broadly effective as a study surface ‚Äî 28 objective pages, 28 per-objective quiz banks with auto-mastery tracking, 5 full practice exams, and a 706-term vocab trainer all work as designed. Two things failed:

1. **Practice-exam state and logs were under-spec'd.** The per-objective quizzes had localStorage persistence, a Reset button, and detailed attempt logs from day one. The 5 practice exams did not ‚Äî state was in-memory only (lost on reload/submit), and log entries recorded *which* questions were missed but not *what was picked* vs *what was correct*. **Fixed on 2026-05-26** for all 5 practice exams (see [¬ß Fix summary](#fix-summary) below).
2. **An AI session on 2026-05-25 destroyed the question banks of 7 per-objective quizzes** by calling `wiki_write` (full-file rewrite) when `wiki_edit` (surgical patch) was the right tool. The pages are not recoverable from inside the wiki; the user needs to restore from git history or Obsidian local trash. **Banners are now in place** on the 7 affected files pointing here.

The biggest forward lever is tighter AI tool discipline: `wiki_write` is for new pages or genuine majority-rewrites only; refactors and small edits always go through `wiki_edit`. The new [[_template/_LLM-OPERATING|_template/_LLM-OPERATING.md]] codifies this.

## What worked

The architecture of the wiki has held up well across roughly a week of build-out plus several editing passes:

- **Transcript layer is immutable and respected.** Nothing in `transcripts/` has been touched since intake ‚Äî exactly the design intent.
- **Objective pages are stable and complete.** All 28 objective pages exist, have distilled content, and have YAML front matter tracking confidence and mastery. The status legend (`seed ‚Üí draft ‚Üí active ‚Üí mature ‚Üí retired`) gives a clear shorthand for where each page is.
- **The quiz scaffold pattern works.** `quizzes/_quiz-scaffold.md` holds the per-objective quiz engine. Individual quiz files contain only the questions and a small bootstrap that loads the scaffold. State persists across reloads via `localStorage`; the Reset button clears it cleanly; the auto-`‚úì` rename on ‚â•90% first-try accuracy is unobtrusive and feels good when it triggers. The shared scaffold also means a single edit to that file propagates to all 28 quizzes ‚Äî exactly the leverage we wanted.
- **Logging discipline (`_log.md`) is invaluable.** Every write is logged with a one-line description and the affected path. This is what made the 2026-05-25 corruption investigation tractable ‚Äî a 30-second `wiki_log_tail` query showed exactly which 7 files were `wiki_write`-overwritten on the same date with near-identical "refactor X to shared scaffold" messages. Without the log this could have stayed silent until each quiz was opened individually.
- **`wiki_edit` is the right primitive for content work.** It is transactional (all-or-nothing on the batch), refuses YAML overlap, requires unique anchors, and auto-bumps `updated:`. Used correctly, it makes catastrophic loss almost impossible.

## What failed

### Failure 1 ‚Äî Practice-exam state was in-memory only

The 5 practice exams (`practice-exam-a` through `practice-exam-e`) were authored as monolithic files (~110‚Äì155 KB each) with the full quiz engine inlined, rather than loading the shared scaffold. That divergence came at a cost. The per-objective scaffold had localStorage persistence and a Reset button from the start; the practice-exam engine had neither. Concretely:

- Picking an answer, navigating away, and coming back wiped the selection.
- Clicking **Grade my test** rendered the report card ‚Äî but reloading the page wiped everything, *including* the graded state.
- The sidecar log entry written on grading recorded question numbers (`- A23 (Obj 2.4) ‚Äî phishing`) but not the user's pick or the correct answer. A miss list was a to-do list, not a learning artifact.

This was reported as the user's complaint #1 ("test results didn't save fully") and complaint #2 ("test cleared when submitted").

### Failure 2 ‚Äî Seven per-objective quizzes were silently corrupted

On 2026-05-25 an AI session ran 7 consecutive `wiki_write` operations with changelog entries of the form *"refactor X to shared scaffold (picks up auto-‚úì rename feature)"*. The intent was to migrate each file from an inlined engine to the shared scaffold loader pattern. The execution was wrong: `wiki_write` replaces the entire page content, and the new content written was only the empty stub (front matter + `_Attempt history is appended to‚Ä¶_` line). The questions array, the bootstrap, the scaffold loader ‚Äî all gone.

The affected files are all marked with the `‚úì` mastery suffix and are the most heavily-trafficked of the per-objective banks:

- `quizzes/1.1-controls ‚úì.md`
- `quizzes/1.3-change-management ‚úì.md`
- `quizzes/1.4-cryptography ‚úì.md`
- `quizzes/2.2-vectors ‚úì.md`
- `quizzes/2.3-vulnerabilities ‚úì.md`
- `quizzes/2.4-malicious-activity ‚úì.md`
- `quizzes/2.5-mitigation ‚úì.md`

The other 21 per-objective quizzes were also "migrated to the shared scaffold" ‚Äî but they were migrated on 2026-05-23 via `wiki_edit` (surgical patches) rather than `wiki_write`, and survived intact. **Same goal, different tool, opposite outcome.** This is the cleanest example of the wiki_write-vs-wiki_edit discipline we'll ever have.

### Failure 3 ‚Äî Broader AI tool discipline

The 2026-05-25 incident was the most visible failure mode, but the same family of issues showed up in subtler ways during the build:

- Multiple sessions read the same page multiple times within a few turns instead of batching with `wiki_read_many`.
- Sessions occasionally rebuilt the contents of a page from scratch in `wiki_write` when a 50-character `wiki_edit` would have done it ‚Äî wasting tokens, increasing risk surface, and slowing iteration.
- Sessions sometimes wrote a `wiki_write` immediately after a `wiki_read` of the same page without confirming what was being preserved.

None of these caused data loss individually. The 2026-05-25 incident was the same class of impatience finally costing real content.

## Root cause

The proximate cause of the 2026-05-25 corruption is: the AI chose `wiki_write` for a task whose correct tool is `wiki_edit`. The 7 quizzes had a small inlined engine that needed to be replaced with a small bootstrap loader ‚Äî a textbook surgical patch.

The contributing factors are tooling-shape, not just judgment:

- **`wiki_write` is too sharp a tool to have available without friction.** It is the right primitive for new pages and for genuine wholesale rewrites. It is the wrong primitive 95% of the time. Yet it sits next to `wiki_edit` in the toolset with no UI distinction.
- **No explicit AI operating guide existed in the wiki.** `STUDY.md` describes the schema (what pages should contain) but did not codify the meta-rules around how to *modify* pages. The 21-survivors-via-`wiki_edit` vs 7-corrupted-via-`wiki_write` outcome is exactly what an operating guide would have prevented.
- **The practice exams diverged from the scaffold pattern.** Monolithic 150 KB files are inherently riskier to edit than 8 KB files that load shared code. Every edit to a practice exam is an edit through more surface area.

## Fix summary

### Practice-exam state and logging (2026-05-26)

Applied to all 5 practice exams (`a`, `b`, `c`, `d`, `e`):

- **State persistence** via `localStorage` under per-exam key `claude-pe-state:practice-exam-<letter>`. MCQ selections, PBQ selections, and the `graded` flag survive reloads.
- **Reset button** at the top of the exam (mirrors the per-objective quiz Reset). Confirms before clearing.
- **Visual restoration** on reload ‚Äî selected MCQ options show as selected, PBQ select dropdowns restore their values. If `graded` was true on load, all controls render disabled.
- **"Already graded" banner** appears when a graded attempt is loaded; the report-card view itself does not auto-restore (avoids duplicate log writes), so the banner directs the user to hit Reset to start fresh.
- **Log content expansion** ‚Äî each missed MCQ now records the user's pick and the correct letter(s) on a separate indented line. PBQ match items record picked-vs-expected. PBQ firewall rules record `field=picked‚Üíexpected` for each incorrect field. The log file (`sessions/quiz-logs/practice-exam-<letter>.log.md`) is now a usable learning artifact rather than a question-number list.

All 5 exams now match the per-objective quiz behavior on the points that mattered. **One residual gap:** the 5 exam files remain monolithic with inlined engines. A future pass should refactor them onto a shared `_practice-exam-scaffold.md` so future fixes propagate via one edit instead of five. The new [[_template/_practice-exam-scaffold|_template/_practice-exam-scaffold.md]] is the canonical version of that engine and is ready to lift in when the refactor is scheduled.

### Corrupted-quiz flags (2026-05-26)

A banner has been prepended to each of the 7 affected quiz pages pointing here for context and to the user's git history / Obsidian local trash for restoration. The banners do not attempt to reconstruct the question content ‚Äî that is unrecoverable from inside the wiki tooling and should be restored from external backup.

### Template wiki (2026-05-26)

A new `_template/` folder contains the canonical starting point for future study wikis on different subjects. See [[_template/README|_template/README.md]]. The template bakes in:

- A generic schema (parameterized version of this wiki's `STUDY.md`)
- The shared quiz scaffold with localStorage persistence + Reset + auto-‚úì mastery
- A shared practice-exam scaffold with all the fixes from this AAR baked in (so future practice exams are small stubs, not 150 KB monoliths)
- `_LLM-OPERATING.md` ‚Äî explicit tool-discipline rules for any AI editing the wiki

## Forward rules

1. **`wiki_write` is for new pages or wholesale rewrites only.** Any task that says "refactor X" or "update Y" or "patch Z" goes through `wiki_edit`. If unsure, prefer `wiki_edit` ‚Äî the worst case is the edit doesn't match and you re-plan, vs the worst case of `wiki_write` which is data loss.
2. **Always `wiki_read` immediately before any `wiki_write` of an existing page.** Confirm what you're preserving. If the new content is shorter than the existing content, that is a red flag ‚Äî read again before committing.
3. **Batch reads with `wiki_read_many` whenever ‚â• 2 pages are needed.** This is process, not preference ‚Äî it reduces token cost and conversation drift.
4. **Check `wiki_log_tail` before any bulk operation.** If another session was active recently on the same area, it might be in a state you don't expect.
5. **Practice-exam-style monoliths are an anti-pattern.** New full-exam content should load `_practice-exam-scaffold` rather than inline the engine.
6. **The shared scaffold is the contract.** Per-objective and practice-exam content files contain *questions plus a small loader*, nothing more. Engine code goes in the scaffold. This makes bug fixes a one-edit operation, and makes content files small enough that `wiki_write` is rarely tempting.

## Recovery plan for the 7 corrupted quizzes

The user holds the recovery options:

1. **Git history** of the vault, if version-controlled. The pre-2026-05-25 commits hold the original question banks.
2. **Obsidian local trash** (`.trash/` folder in the vault) ‚Äî Obsidian retains a per-deletion history of overwritten files for a configurable window.
3. **Manual reconstruction** as a last resort ‚Äî the corresponding objective pages have the source material, but writing fresh questions is significant work.

Once content is restored, the banners can be removed via `wiki_edit` and a brief note added to `_log.md` confirming recovery.

## Open questions

- Should the auto-`‚úì` mastery rename, currently per-quiz, also apply to practice exams once a passing threshold is hit? Today the practice exams have no `‚úì` suffix mechanic.
- The 5 practice exams have inconsistent display labels ‚Äî exam D shows `B1-B5` PBQ labels (a copy-paste error from when it was forked from exam B) while logging entries with `D` prefix. Cosmetic, not a study blocker, but worth a future cleanup pass.
- Should `wiki_write` require an explicit `force_full_rewrite=true` flag at the tool level to prevent accidents? That's a wiki-tooling change, not something this AAR can land on its own.
